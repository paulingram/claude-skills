## Overview

The architect-team plugin is a single-codebase Claude Code plugin (no peer codebases in scope for this orchestration run). It does NOT have inter-codebase service-to-service calls, shared databases, queues, or contract files. Its "integrations" are entirely external — it composes other Claude plugins, an external CLI (openspec), Python testing tools, Playwright + chromium, and (v0.9.18, opt-in) two email-delivery services — Gmail SMTP and the SendGrid API — engaged by the per-project notifier only when a project supplies `.architect-team-notify.json`.

This INTEGRATION_MAP.md documents those external integrations so future orchestrator runs against the same workspace can short-circuit Phase −1C via the freshness check.

The Docker provisioning includes a pre-flight `docker rm -f` to handle name collisions from prior failed teardowns.

### Cross-session lock layer (v1.0.0)

The user has been asking for cross-session parallelism — *"can I run multiple `/architect-team` invocations at once?"* — since v0.9.x. Given the Agent Teams docs' "one team at a time per Lead" + "no nested teams" constraints, in-session parallelism is impossible; parallelism IS via separate Claude Code sessions. v1.0.0 ships the coordination primitive that makes this safe: **`.architect-team/locks/<scope-hash>.json`**.

**Lock-file shape:**

```jsonc
{
  "holder": "run-<slug>",                   // human-readable run identifier
  "scope_glob": "src/auth/**",              // the file-scope claim
  "acquired_at": "2026-05-28T01:42:00Z",    // ISO 8601 UTC timestamp
  "ttl_seconds": 14400,                      // default 4h (14400s)
  "run_id": "<run-uuid>"                     // unique per /architect-team invocation
}
```

**The four primitives** (all in `hooks/locks.py`, stdlib-only):

| Function | Behavior |
|---|---|
| `acquire_lock(scope_glob, ttl_seconds, run_id)` | Hashes the scope_glob to a filename; if no non-stale lock at that path intersects, writes a new lock file and returns the lock id. If an intersecting non-stale lock exists, returns `blocked: <existing-lock-id>` (the caller's choice whether to queue or surface a conflict). Malformed / missing-field lock files are treated as stale and auto-released. |
| `release_lock(lock_id)` | Removes the lock file. Idempotent — releasing a non-existent lock is a no-op. |
| `detect_stale()` | Returns the lock ids whose `acquired_at + ttl_seconds` is in the past, or whose JSON is malformed / missing required fields. Used by `acquire_lock` internally; can be called standalone for cleanup. |
| `globs_intersect(a, b)` | Returns True when two path-globs could match a common path. Reuses the non-overlapping-file-scope discipline from `team-spawning-and-review-gates`. |

**Use case — two `/architect-team` invocations in separate Claude Code sessions:**

- **Session 1** (Lead A): calls `acquire_lock(scope_glob="src/auth/**", ttl_seconds=14400, run_id="run-A")` → writes the lock at `.architect-team/locks/<hash-of-auth>.json` with `holder: "run-A"`, returns the lock id. Lead A proceeds to dispatch teammates against `src/auth/**`.
- **Session 2** (Lead B, started 10 minutes later): calls `acquire_lock(scope_glob="src/auth/login/**", ...)` → `globs_intersect("src/auth/**", "src/auth/login/**")` returns True → returns `blocked: <Lead-A's-lock-id>`. Lead B's choice: queue (poll until the lock releases or goes stale) or surface a conflict to the user.
- **Disjoint case:** Session 2 calls `acquire_lock(scope_glob="src/billing/**", ...)` → `globs_intersect("src/auth/**", "src/billing/**")` returns False → acquire succeeds, both leads run truly parallel.
- **Crash case:** Lead A crashes; its lock sits at `.architect-team/locks/`. After 4 hours its `acquired_at + ttl_seconds` is in the past; the next intersecting `acquire_lock` invokes `detect_stale()`, removes the stale lock, and succeeds. No human cleanup required.

The lock layer is BEST-EFFORT cross-session coordination, NOT a hard-real-time mutex. If two leads in two sessions both call `acquire_lock` within the same wall-clock millisecond, the filesystem semantics of `os.replace` / atomic write decide the winner; the loser's call detects the winner's lock on its next read and returns `blocked`. The layer guarantees no data corruption and no silent overlap; it does NOT guarantee a strict FIFO acquisition order under perfect concurrency. Acceptable for the user's actual use case (one ad-hoc parallel run per developer per workspace).

This handoff IS the structural primitive that opens cross-session parallelism. Update `hooks/locks.py` in lockstep with the discipline documented in `skills/team-spawning-and-review-gates/SKILL.md`'s non-overlapping-file-scope section.

## Runtime dependencies (v1.1.0)

### Teams-mode dependencies (v1.0.0)

v1.0.0 adds a NEW DEPENDENCY for teams-mode dispatch: **`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`** — set in the shell environment OR as `{"env": {"CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"}}` in `~/.claude/settings.json` — AND **Claude Code ≥ 2.1.32**. Both are required for teams mode; without either the pipeline silently falls back to subagents mode (the v0.10.0 behavior, unchanged). The mode-detection helper `scripts/setup/teams_mode.py::is_teams_mode_available()` makes the decision once at pipeline startup; the decision is recorded in `intake-state.json` and the pipeline never re-checks. `/architect-team-setup` (`scripts/setup/setup.py`) checks both at install time and offers to add the env var to `~/.claude/settings.json` with explicit user consent. The `--no-teams` flag on any of the three pipeline commands (`/architect-team`, `/architect-team:bug-fix`, `/architect-team:mini`) forces subagents mode even when teams are available.

### Worktree-aware state resolution (v1.1.0)

v1.1.0 introduces a new INTERNAL runtime dependency (not user-facing — no env var, no config, no opt-in): **the lock layer and the MemPalace integration both resolve their root path via `scripts/setup/worktree_paths.py::shared_state_dir()`**, which routes through `git rev-parse --git-common-dir` to identify the MAIN worktree's `.architect-team/` directory. This closes the v1.0.0 gap where two `/architect-team` sessions in two git worktrees of the same repo each got their own isolated locks dir + their own isolated MemPalace, defeating the cross-session coordination intent.

**Resolution semantics** (canonical, applied by every consumer):

| Function | Returns | Used for |
|---|---|---|
| `shared_state_dir() -> Path` | parent(`git rev-parse --git-common-dir`) / `.architect-team` (or cwd / `.architect-team` if git probe fails) | `locks/` (via `hooks/locks.py` default), `.mempalace/palace/` (via `mempalace-integration` skill), `run-history/` |
| `run_state_dir() -> Path` | `Path.cwd() / '.architect-team'` (always per-worktree) | `reviews/`, `teammates/`, `handoffs/`, this-run's `openspec/changes/<slug>/`, this-run's findings + refined-prompts |
| `is_worktree() -> bool` | True iff `git rev-parse --git-dir` != `git rev-parse --git-common-dir` (absolute-path comparison) | downstream callers that want to know |

**The 3-layer model** the helper enables (canonical, applied to cross-session coordination overall):

| Layer | Concern | Mechanism | Resolution |
|---|---|---|---|
| Filesystem isolation | Two sessions can't clobber each other's working tree, index, or branch | Git worktrees (`git worktree add`) | Each session checks out its own worktree on its own branch — use `superpowers:using-git-worktrees` for lifecycle |
| Architectural coordination | Two sessions can't both decide to refactor the same scope-glob | `.architect-team/locks/` JSON locks (v1.0.0) | `shared_state_dir() / 'locks'` — both worktrees see the same lock dir |
| Context sharing | Session B sees what session A produced | MemPalace + CLAUDE.md + codebase maps | `shared_state_dir() / '.mempalace' / 'palace'` — both sessions share one palace |

**Backwards compatibility:** single-session users (no worktrees) see ZERO behavior change. `shared_state_dir()` and `run_state_dir()` degenerate to the same path in a non-worktree clone. The lock layer reads/writes the same location it did in v1.0.0. No env var, no flag, no opt-in. The lock layer's explicit `locks_dir=` parameter (used by `tests/test_locks.py` for test isolation) is preserved verbatim — all 17 v1.0.0 lock tests pass unchanged after the v1.1.0 default-resolution change.

**Failure modes:** every git subprocess invocation in `worktree_paths.py` is wrapped to fall back to `Path.cwd() / '.architect-team'` on any failure (not a git repo, git missing, broken metadata, empty stdout). `hooks/locks.py`'s `_default_locks_dir()` further wraps the helper import in a try/except — if `worktree_paths.py` is unreachable for any reason, the lock layer falls back to the legacy `DEFAULT_LOCKS_DIR` constant, so the public lock API never raises. The cross-worktree coordination guarantee degrades in these edge cases (each worktree's lock layer becomes isolated again), but no run is interrupted.

**Out of scope for v1.1.0:** cross-CLONE coordination (two separate clones of the same project on disk do NOT coordinate via this layer — the lock files + MemPalace are repo-local, not machine-wide); worktree lifecycle automation from inside the pipeline (the `superpowers:using-git-worktrees` skill already handles this).

## Deployment Topology

The architect-team plugin deploys as a Claude Code plugin via:

1. User runs `/plugin marketplace add https://github.com/paulingram/claude-skills.git`. The harness git-clones to `~/.claude/plugins/marketplaces/architect-team-marketplace/`.
2. User runs `/plugin install architect-team@architect-team-marketplace`. The harness copies the plugin into `~/.claude/plugins/cache/architect-team-marketplace/architect-team/<version>/` and registers it in `~/.claude/plugins/installed_plugins.json`.
3. User runs `/reload-plugins`. The harness reads `.claude-plugin/plugin.json`, `hooks/hooks.json`, every `skills/<name>/SKILL.md`, every `agents/<name>.md`, and every `commands/<name>.md` and registers them as namespaced (`architect-team:*`).
4. User runs `/architect-team:architect-team-setup` once to install external deps (openspec / pytest / playwright / chromium).
5. User invokes `/architect-team:architect-team <path>` to run the pipeline.

There is no server-side deployment. All execution is local in the Claude Code session.

## Failure Modes (cross-system)

- **Missing prerequisite plugin (superpowers / cartographer / ralph-loop)**: setup script reports `[missing]` with the install command. Orchestrator skill bodies reference these by name and would fail at first invocation if absent.
- **Stale marketplace clone**: pre-v0.2.2 manual `git pull` was needed because `/plugin marketplace add` did not re-fetch on remove+re-add. Post-v0.2.2 confirmed: `/plugin marketplace update <name>` does fetch and reports "1 plugin bumped" when a new commit is on origin.
- **Hook over-aggressive**: pre-v0.2.2 the review-gate hook blocked every TaskUpdate. v0.2.2 scoped it to teammate tasks only.
- **`uv pip install` without venv**: pre-v0.1.1, setup script failed to install Playwright on machines with `uv` on PATH. v0.1.1 detects venv and adds `--system` when none active.
- **Skill name collision with command**: pre-v0.2.0, the skill named `architect-team` collided with the slash command of the same name; Skill tool resolved to the command body. v0.2.0 renamed skill to `architect-team-pipeline`.
- **`disable-model-invocation: true` on a skill the command needs to invoke**: pre-v0.2.1, the orchestrator skill had this flag set, blocking the Skill tool. v0.2.1 removed it.

## Architectural notes & inherent limits

The v0.2.3-era open questions (REQ-001 `$ARGUMENTS` propagation, REQ-002 path-traversal sanitization, REQ-003 test-coverage gaps, REQ-004 hook-rejection escalation policy, REQ-005 spec drift) are all **resolved** — see `CHANGELOG.md`. The v0.9.9 logic-implementation review surfaced and closed three tiers of holes. What remains are the inherent limits of an LLM-agent architecture — recorded here honestly rather than papered over:

- **The orchestrator cannot be hooked mid-run.** It is the main session; no hook gates its mid-run behaviour. Every phase discipline is trust-based Markdown. The mitigations are real but bounded: the `Stop` hook (`pipeline-completion-audit.py`) gates the orchestrator's *terminal* state, and the Phase 8 `--check` gate makes "clean pass" a checked fact — but neither can enforce "did you run Phase 5's reviews."
- **The automated test suite is structural, not behavioural.** The 647 pytest tests prove the plugin's Markdown / JSON / hook scripts are well-formed and that components do not drift apart (`test_cross_consistency.py`) — and, for the notifier, that its config/provider/dispatch/CLI logic behaves correctly with the genuinely-external transports (`smtplib.SMTP`, `urllib.request.urlopen`) mocked. They do NOT prove the live multi-agent pipeline produces correct software — behavioural / integration testing of the orchestration is outside an automated suite by nature.
- **Runs are non-deterministic.** Re-running the pipeline on the same input yields different teammates, classifications, and convergence. The evidence-artifact trail + the Stop-hook audit make outcomes *auditable*, not *reproducible*.
- **Single-codebase 3-explorer convergence is degenerate** but the skill still mandates running it (it must produce the INTEGRATION_MAP.md that `reuse-first-design` consults). Accepted as-is.
