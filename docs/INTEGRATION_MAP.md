The CLAUDE TEAM SIX plugin (internal slug `architect-team`) is a single-codebase Claude Code plugin (no peer codebases in scope for this orchestration run). It has NO inter-codebase service-to-service calls, shared databases, queues, or contract files between peer repos. Its integration surface is of two kinds, and this map documents both:

1. **External integrations** (`## Per-Pair Integration`) — the plugins / CLIs / services it composes at runtime: superpowers (a HARD dependency, v3.9.0), cartographer, ralph-loop, the openspec CLI, pytest, Playwright + chromium, the optional MemPalace memory layer, the optional opt-in Gmail-SMTP / SendGrid email notifier, and test-scoped Mailpit. **No new external integration has been added since v2.4.0** — `scripts/notify/notify.py` (Gmail SMTP + SendGrid) remains the ONLY genuinely external egress of the plugin core, and it is opt-in + best-effort. The plugin core stays stdlib-only.
2. **Internal tier integration** (`## Internal tier integration (v3.30.0 synthesis)` + the deep-dive sections below) — the seams BETWEEN the plugin's own tiers: skills ↔ agents ↔ commands, the hooks that gate them, the `scripts/` deterministic engines the skills contract, the `services/` adapter seams, the v3.30.0 run-continuity marker, and the on-disk state schemas that carry data between phases and sessions. This is a documented invocation contract carried in the Markdown bodies, enforced deterministically only at the hook seams.

Current inventory (v3.38.0): **48 skills, 39 agents, 23 commands, and 7 wired hook enforcement scripts across 8 hook events**, plus the `scripts/` + `hooks/` deterministic-engine surface (now including the v3.31.0 `scripts/compliance/instruction_compliance.py` instruction-compliance lint, contracted by `docs/INSTRUCTION_COMPLIANCE_RUBRIC.md` + the enforced `tests/test_instruction_compliance.py` gate, and the v3.34.0 ten-event `scripts/notify/notify.py` informative-run-notifications surface wired into all four pipelines) and the `services/` server tier (stdlib-only core). As of **v3.32.0** all 39 agents default to `model: fable` (Fable 5) with an implemented Opus 4.8 fallback — `services/common/service_config.py::resolve_model` (routed through `build_llm_client`) + the `scripts/setup/set_default_model.py` frontmatter lever — and `scripts/setup/setup.py` carries the first-install hardening (cartographer marketplace provenance `kingbootoshi/cartographer`, an npm EACCES `--prefix ~/.local` retry, the PEP-668 `uv`→`pip --user`→`--break-system-packages` install ladder + `tiktoken`, and `--yes`/`CT6_SETUP_ASSUME_YES` non-interactive consent). This INTEGRATION_MAP.md documents both integration surfaces so future orchestrator runs against the same workspace can short-circuit Phase −1C via the freshness check.

## Per-Pair Integration

The architect-team plugin does not integrate with any peer codebase. It DOES depend on the following external systems, all listed for transparency:

### architect-team ↔ superpowers plugin

- **Protocol:** in-process Skill tool invocation (architect-team doesn't directly invoke superpowers, but expects the user to have superpowers installed; many recommended subagent flows reference superpowers' methodology in their bodies).
- **Endpoint:** `/plugin install superpowers@claude-plugins-official`.
- **Required because (v3.9.0):** superpowers is a **HARD dependency that is actually exercised at runtime**. Every pipeline body (`architect-team-pipeline` / `bug-fix-pipeline` / `mini-architect-team-pipeline` / `ux-test-builder`) opens with a `## Plugin prerequisites (v3.9.0)` **superpowers pre-flight abort gate** and weaves concrete `superpowers:brainstorming` / `test-driven-development` / `systematic-debugging` / `verification-before-completion` invocations through its phases — replacing the prior decorative "Superpowers-driven teammates" framing. Canonical home: `common-pipeline-conventions` `## Uniform plugin usage (v3.9.0)`. Hard-blocking governs plugin PRESENCE only; user `CLAUDE.md` / `AGENTS.md` instructions still win.
- **Failure mode (v3.9.0):** superpowers is a HARD dependency — `setup.py` **HARD-blocks (exit 1)** on its absence (no longer a warning). The vendored `openspec-propose` authoring skill is verified as a **4th HARD-gated prerequisite** via `ensure_openspec_propose_skill()` (a missing vendored `.claude/skills/openspec-propose/SKILL.md` — or an opsx/openspec plugin id — is also exit 1).

### architect-team ↔ cartographer plugin

- **Protocol:** in-process Skill tool invocation (`architect-team:intake-and-mapping` invokes `cartographer:cartographer` as a sub-skill in Phase −1B).
- **Endpoint:** `/plugin install cartographer@cartographer-marketplace`.
- **Required because:** cartographer produces `<codebase>/docs/CODEBASE_MAP.md` which the orchestrator depends on for Phase −1B and many downstream phases (reuse-first design, system-architect agent).
- **Failure mode:** if cartographer absent, Phase −1B cannot produce CODEBASE_MAP.md → orchestrator must surface this as a blocker. `setup.py` reports missing.

### architect-team ↔ ralph-loop plugin

- **Protocol:** slash-command invocation (`architect-team:intake-and-mapping` describes Phase −1B/−1C wrapping in `/ralph-loop "<prompt>" --completion-promise "<exit string>"` — loops until the promise; no iteration cap per v3.8.0 unbounded solving).
- **Endpoint:** `/plugin install ralph-loop@claude-plugins-official`.
- **Required because:** the canonical Phase −1B/−1C iterate-until-converged loop is implemented via ralph-loop.
- **Failure mode:** if ralph-loop absent, the orchestrator must implement the iteration logic natively (current behavior in this dogfood run since `/ralph-loop` is a slash command not directly invokable from inside an orchestrator context — no functional break, just less ceremony).

### architect-team ↔ @fission-ai/openspec CLI

- **Protocol:** subprocess (Bash) invocation of `openspec` binary.
- **Endpoint:** `npm install -g @fission-ai/openspec@latest` (run by `setup.py`).

- **OpenSpec authoring skill (v3.9.0 — 4th HARD prerequisite):** the vendored `openspec-propose` skill (the Phase 0 authoring playbook) is verified by `ensure_openspec_propose_skill()`; a missing vendored `.claude/skills/openspec-propose/SKILL.md` (or an opsx/openspec plugin id) is an exit-1 HARD block in `setup.py`.

### architect-team ↔ pytest + pytest-asyncio + httpx

- **Protocol:** subprocess invocation; tests imported as Python modules.
- **Endpoint:** pip-installed via `setup.py` (uses `uv pip install --system` when no venv per the v0.1.1 fix; falls back to plain pip otherwise).
- **Required because:** plugin self-tests use pytest; backend teammates use httpx for dev-API integration tests.

### architect-team ↔ Playwright + chromium

- **Protocol:** Playwright Python package + chromium browser binary.
- **Endpoint:** `pip install playwright && playwright install chromium` (run by `setup.py`).
- **Required because:** the `integration` agent, the `frontend` teammate, and the `visual-capture` / `visual-analyzer` agents use Playwright for user-flow tests and live-app rendering against the dev environment.
- **Failure mode:** if Playwright or chromium absent, no Phase 5 frontend integration tests or visual-verification renders can run. `setup.py` checks both Python package import AND chromium cache directory presence (the v0.1.1 fix).

### architect-team ↔ MemPalace (optional, v0.9.4)

- **Protocol:** subprocess (Bash) invocation of the `mempalace` CLI; optionally an MCP server (`mempalace-mcp`) registered in Claude Code.
- **Endpoint:** `uv tool install mempalace` (or `pip install --user mempalace`), run by `/architect-team:mempalace-install` → `scripts/setup/install_mempalace.py`.
- **Required because:** the pipeline auto-mines every artifact (maps, RCAs, diagnostic plans, SRs, coverage maps, final reports) into a per-workspace MemPalace store at `<workspace>/.mempalace/palace`, and named agents search it for prior context before producing output (per `mempalace-integration`).
- **Failure mode:** MemPalace is an ergonomics layer, NOT a hard dependency. If it is not installed, every wake-up / mine / search is skipped with a one-line note and the pipeline proceeds without cross-run memory. Graceful degradation by design.

### architect-team ↔ Gmail SMTP (optional, opt-in notifier, v0.9.18)

- **Protocol:** SMTP over STARTTLS via the Python standard library — `smtplib.SMTP` + `email.message.EmailMessage` + `ssl` (a 15-second bounded socket timeout). No third-party dependency.
- **Endpoint:** `smtp.gmail.com:587`. Engaged only when a target project supplies `.architect-team-notify.json` with `provider: "gmail"`; the notifier `scripts/notify/notify.py` (the `GmailProvider`) authenticates with a Gmail **app password** read from the environment variable named in config (`gmail.app_password_env`).
- **Required because:** the pipeline emits opt-in per-project email notifications at ten events (`run_start` — the kickoff email embedding the architecture + solution plan via `--plan-file` — `phase_start`, `phase_complete`, `waiting_on_agents`, `agents_complete`, `issue_discovered`, `git_commit`, `deploy`, `run_complete`, `heartbeat`) when a project opts in (per `project-email-notifications` REQ-002/003/005 + the v3.34.0 informative-run-notifications overhaul; wired in all four pipelines incl. ux-test).
- **Failure mode:** strictly best-effort. A missing config, a missing/empty `gmail.app_password_env` variable, an SMTP auth/connection/send error — each is caught, written to stderr as a single warning (never echoing the secret), and the notifier exits 0. A notification failure can never block, fail, or alter a pipeline run; with no `.architect-team-notify.json` the notifier is a silent no-op.

### architect-team ↔ SendGrid v3 mail-send API (optional, opt-in notifier, v0.9.18)

- **Protocol:** HTTPS POST via the Python standard library — `urllib.request` with a JSON body (`personalizations` / `from` / `subject` / `content`) and a `Bearer` Authorization header (a 15-second bounded socket timeout). No third-party dependency (the official `sendgrid` SDK was deliberately not adopted — stdlib-only, per design D3).
- **Endpoint:** `https://api.sendgrid.com/v3/mail/send`. Engaged only when a target project supplies `.architect-team-notify.json` with `provider: "sendgrid"`; the notifier `scripts/notify/notify.py` (the `SendGridProvider`) authenticates with the SendGrid API key read from the environment variable named in config (`sendgrid.api_key_env`).
- **Required because:** the alternate email provider for the opt-in per-project notification feature (per `project-email-notifications` REQ-002).
- **Failure mode:** strictly best-effort. A missing config, a missing/empty `sendgrid.api_key_env` variable, a non-2xx HTTP response, a network error — each is caught, written to stderr as a single warning (status code only, never the response body or the secret), and the notifier exits 0. A notification failure can never block, fail, or alter a pipeline run.

### architect-team ↔ Mailpit (test-scoped email capture, v0.9.34)

- **Protocol:** Docker container (`axllent/mailpit`) exposing SMTP on port 1025 and a REST API on port 8025. When Docker is unavailable, falls back to a direct binary download from `github.com/axllent/mailpit/releases/latest/download/` (Linux amd64, macOS amd64, Windows amd64). The REST API is accessed via `fetch()` inside Playwright test specs (no SDK, no third-party dependency).
- **Endpoint:** `http://localhost:8025/api/v1/search?query=...` for server-side email search (v0.9.35 refinement — replaces the prior `GET /api/v1/messages?limit=10` client-side filter), `GET /api/v1/message/{id}` for full message retrieval, `DELETE /api/v1/messages` for pre-test cleanup. SMTP trap listens on `localhost:1025`.
- **Required because:** the `email-testing` skill (v0.9.34) provides automatic Mailpit-based email flow verification across all QA agents (`bug-replicator`, `flow-executor`, `integration`). When any QA agent detects email-touching work in its slice, it provisions Mailpit, reroutes the dev environment's SMTP through it, triggers the email send via Playwright UI interaction, captures the email via the REST API, extracts and classifies every link by purpose, and follows every link in Playwright to complete the full user flow.
- **Failure mode:** test-scoped and fault-isolated. Mailpit provisioning failure exits as `env-failure` for the email portion of the test — it does NOT block non-email tests in the same slice. Teardown is mandatory (`try/finally`) — a dangling SMTP trap would intercept emails in subsequent test runs. The Docker provisioning includes a pre-flight `docker rm -f` to handle name collisions from prior failed teardowns.

### architect-team ↔ Claude Design MCP (`claude_design`, optional, v3.33.0)

- **Protocol:** the `claude_design` MCP tools, loaded at RUNTIME via ToolSearch. The deterministic engine `scripts/claude_design/claude_design_import.py` NEVER calls the network itself — the real fetch is INJECTED into it as a `ClaudeDesignSource` adapter (offline tests use `FakeClaudeDesignSource`). No SDK is bundled; no MCP tokens or credentials are persisted.
- **Endpoint:** `https://api.anthropic.com/v1/design/mcp`; authentication via `/design-login`. A project is addressed by a `claude.ai/design/p/<id>` link (optionally a `?file=<screen>` focus + a trailing `Implement: <path>` target).
- **Required because:** the v3.33.0 `claude-design-import` capability (the `claude-design-import` skill + the engine) detects a Claude Design offer at Phase −1 intake, fetches the WHOLE project through the MCP, materializes it path-safely to `<workspace>/.architect-team/claude-design/<project-id>/`, and hands that directory to the EXISTING `oracle-deriver` interactive-mockup oracle path (then `interactive-mockup-discovery` then `visual-to-api-design`) — an ADDITIVE design-input source alongside the existing local/zip discovery, with no new downstream consumer.
- **Failure mode:** OPTIONAL + graceful. When the MCP is not connected (or `/design-login` has not run), `plan_when_unavailable` returns an instruct-then-fallback plan — instruct the user to connect the MCP + run `/design-login`, and on the user declining auto-fall-back to the existing zip/local design-input path; a run NEVER dead-ends on an unavailable MCP. Design + a runnable stdlib core + tests, NOT a live-deployed integration — the MCP fetch is an adapter boundary.

## Internal tier integration (v3.30.0 synthesis)

This section is the tier-integration synthesis the pre-v3.30.0 map under-documented: how the plugin's own tiers (skills / agents / commands / hooks / `scripts/` engines / the `services/` tier / MemPalace / on-disk state) wire to each other. The wiring is not a call graph in compiled code — it is a documented invocation contract carried in the Markdown bodies, enforced DETERMINISTICALLY only at the hook seams (see the hook map below). Every claim here was read first-hand from boundary code (`hooks/hooks.json`, `hooks/run_continuity.py`, `hooks/pretool_skill_gate.py`, `hooks/pipeline-completion-audit.py`, `hooks/sessionstart-run-continuity.py`, `services/separation.py`, `services/common/service_config.py`); the deep-dive sections that follow (`## Pipeline-to-pipeline handoffs`, `## Interactive-mockup-discovery flow`, `## Runtime dependencies`, `## Code & Data Lineage Graph (CDLG) integration`, `## Contracts & Schemas Catalog`) carry the per-surface detail.

### Skills ↔ agents ↔ commands wiring

Three registered surfaces, all namespaced `architect-team:*` by the harness at `/reload-plugins` from `.claude-plugin/plugin.json` + the directory layout: **48 skills** (`skills/<name>/SKILL.md`), **39 agents** (`agents/<name>.md`), **23 commands** (`commands/<name>.md`). They compose in one direction of authority:

- **Commands are thin wrappers.** The five pipeline-DRIVING commands — `/architect-team`, `/architect-team:bug-fix`, `/architect-team:ux-test`, `/architect-team:mini`, `/architect-team:refine-prompt` — each hand off to a **Skill** (these five are the set the PreToolUse skill-gate is scoped to). The FOUR full-pipeline drivers among them (`/architect-team`, `/architect-team:bug-fix`, `/architect-team:ux-test`, `/architect-team:mini`) additionally do a small amount of documented pre-Skill setup in `Bash` (the dispatch banner via `scripts/setup/teams_mode.py::format_dispatch_banner()`, merged-worktree cleanup, auto-worktree creation via `scripts/setup/worktree_lifecycle.py`) before that hand-off; `/architect-team:refine-prompt` does NOT — `commands/refine-prompt.md` carries none of that Bash setup (0 grep hits) and hands straight to its refinement skill. The command body INSTRUCTS the model to invoke the skill; nothing in the command forces it — that force is the PreToolUse gate (hook map below). The other 18 commands are utility / inspection (`/status`, `/memory`, `/discipline-status`, `/closeout`, `/logit`, `/librarian-install`, `/inject`, `/cleanup-worktrees`, …) and mandate no pipeline skill.
- **Skills are the playbooks + LLM-judgment contracts.** The four run-driving playbooks (`architect-team-pipeline`, `bug-fix-pipeline`, `mini-architect-team-pipeline`, `ux-test-builder`) orchestrate phases and dispatch agents. The discipline skills — `common-pipeline-conventions` (the single canonical home for the cross-cutting disciplines: scope, git, unbounded solving, appearance-change policy, and the v3.30.0 run-continuity discipline), plus `team-spawning-and-review-gates`, `documentation-currency`, the `*-mapping` family, `verified-agent-output`, etc. — are referenced BY the playbooks as canonical rule homes.
- **Agents are the workers a skill spawns.** Producer/checker is enforced at spawn level (v0.9.13): an implementer (`backend` / `frontend`) is paired with an independent `task-reviewer` + an `adversarial-reviewer` writing into the SAME evidence file; the ×3 convergence agents (`integration-explorer`, `structure-analyst`, `domain-researcher`) converge round-robin, then a synthesizer produces the canonical artifact (this map is the `integration-explorer` ×3 + `master-synthesizer` output). As of v3.30.0 every spawn brief's FIRST LINE must carry the `[CT6-TEAMMATE <name> RUN <id>]` token (mandated by `team-spawning-and-review-gates`) — the signal both run-continuity guards use to stand down for the pipeline's own workers.

The command→skill routing table is `COMMAND_TO_SKILLS` in `hooks/skill_invocation_audit.py` — the single map REUSED by both the Layer-6 after-the-fact skill-invocation auditor AND the v3.15.0 real-time PreToolUse gate. `EXPECTED_SKILLS` / `EXPECTED_AGENTS` / `EXPECTED_COMMANDS` are the drift pins in the test suite; `tests/test_agents.py` / `test_skills.py` / `test_commands.py` are the Markdown-frontmatter schemas. Agent boilerplate is kept byte-identical across all 39 files by `scripts/setup/sync_agent_boilerplate.py` (a drift-guard test pins it).

The dispatch primitive is Claude Code's experimental **Agent Teams** (long-lived named teammates, a shared task list, direct `SendMessage`, a Lead that owns coordination), gated on `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` + Claude Code ≥ 2.1.32, silently falling back to subagents mode otherwise (`--no-teams` forces it). The decision is made once by `teams_mode.py::is_teams_mode_available()` and recorded in `intake-state.json` (see `## Runtime dependencies`).

### Hooks enforcement seam map (8 events → 7 scripts)

`hooks/hooks.json` is the single registration point the harness reads. It wires **8 distinct hook events** to **7 enforcement scripts**; THREE scripts are each wired to two events. Every command string uses the detect-once polyglot `$(command -v python3 || command -v python) "…"` so a blocking hook runs exactly once and a python3-only host never drops the block.

| Event (matcher) | Script | What it gates |
|---|---|---|
| `PreToolUse[*]` | `pretool_skill_gate.py` | Arm 1: blocks build/dispatch tools until the mandated Skill is invoked. Arm 2 (v3.30.0): the sticky run arm (see run-continuity below). |
| `PreToolUse[Edit]` / `[Write]` / `[NotebookEdit]` | `pretool_unilateral_override_guard.py` | Blocks a unilateral override of a governance file without the sanctioned marker. |
| `PostToolUse[TaskUpdate]` | `review-gate-task.py` | On a teammate task marked complete, requires a valid schema-v7 evidence file (self-review + an independent `task-reviewer` block, reviewer ≠ teammate). |
| `PostToolUse[Skill]` | `pretool_skill_gate.py` | v3.30.0: `record_engagement()` — ENGAGES the `active-run.json` marker the moment a run-driving Skill actually RAN (PostToolUse fires only after the tool executed, so a denied / errored call never engages). |
| `TaskCompleted[*]` | `review-gate-task.py` | Same evidence gate on the completion event. |
| `SubagentStop[*]` | `teammate-idle-check.py` | Idle/complete teammate → evidence + manifest check. |
| `TeammateIdle[*]` | `teammate-idle-check.py` | Teams-mode idle → same. |
| `Stop[*]` | `pipeline-completion-audit.py` | Terminal-state worklist audit (open SRs / editability / test-completeness / visual-fidelity / master-review / an independent `openspec validate --all --strict` re-run / docs-currency / bug-fix witnesses) + the v3.30.0 continuation guard (see run-continuity below). |
| `PreCompact[*]` | `precompact-closeout.py` | v3.18.0 (the FIRST PreCompact wiring): before compaction, runs `hooks/closeout_check.py` and injects a non-blocking doc-currency reminder. |
| `SessionStart[*]` | `sessionstart-run-continuity.py` | v3.30.0 (the FIRST SessionStart wiring): on startup/resume/clear/compact, injects the resume-via-Skill directive when a run is active. |

**The three dual-wired scripts:**

- `pretool_skill_gate.py` — `PreToolUse[*]` (the gate check) + `PostToolUse[Skill]` (marker engagement); its `main()` branches on `payload["hook_event_name"]`. This is how a deterministic hook, with no LLM cooperation, writes the run-continuity marker.
- `review-gate-task.py` — `PostToolUse[TaskUpdate]` + `TaskCompleted[*]`; the same evidence gate on either shape.
- `teammate-idle-check.py` — `SubagentStop[*]` + `TeammateIdle[*]`; branches on payload shape.

Load-bearing facts (verified in `hooks/hooks.json` + the scripts):

- **The orchestrator IS the main session; no hook gates its MID-run behavior.** The hooks gate the EDGES — teammate task completion (`review-gate-task`), teammate idle (`teammate-idle-check`), and the orchestrator's terminal state (`pipeline-completion-audit` Stop). The v3.30.0 run-continuity work is precisely an attack on the two remaining edge gaps: an orchestrator stopping mid-run, and a resumed session hand-solving instead of re-loading the skill.
- **Every script fails open.** Any missing transcript, malformed payload, unavailable substrate module, or unexpected exception returns exit 0 (allow). The kill-switch `CT6_RUN_CONTINUITY_DISABLED=1` disables every run-continuity surface without touching the legacy worklist audit.
- **`hooks/run_continuity.py` is NOT itself a wired hook** — it is the shared substrate the three run-continuity surfaces import, plus the orchestrator-facing CLI. `hooks/skill_invocation_audit.py`'s own Stop-auditor is likewise not wired; the PreToolUse gate reuses its `find_skill_requests` + `COMMAND_TO_SKILLS` detection.

Two carried-forward hook↔agent contracts (detail in `## Contracts & Schemas Catalog`):

- **Review-gate evidence (schema v7)** — `hooks/review_evidence_schema.py` (`SCHEMA_VERSION = 7`, 17 required fields incl. the 5 VAO fields + 3 optional fields `interactions_honored_review` / `live_verification_review` / `appearance_scope_review`). The teammate writes the self-review; the independent `task-reviewer` writes the `independent_review` block; BOTH `review-gate-task.py` and `teammate-idle-check.py` import the shared module and reject `independent_review.reviewer == teammate`, block any `*_review: fail`, and require the `_note` on `n/a`.
- **Layer-3 VAO tools** — the 20 deterministic `verify-*` subcommands dispatched through the `hooks/vao_tools.py` facade over the `hooks/vao/` package (the facade re-exports the full 125-name surface + a byte-identical CLI). The `adversarial-reviewer` (Layer 2) invokes them at Phase 3 against a teammate's diff + tool-log; each writes a deterministic verdict JSON under `.architect-team/vao-verdicts/`. The schema-v7 `*_review` dict-shape CITES the verdict path, and the hook trusts the cited verdict file, not the agent's prose.

### The v3.30.0 run-continuity marker lifecycle

The load-bearing new internal contract surface. A single lifecycle marker — `<cwd>/.architect-team/active-run.json` (`hooks/run_continuity.py`, `MARKER_FILENAME`; `{schema:1, status: active|complete|stood-down, skill, session_id, started_at, updated_at, run_id, slug, phase, completed_at, stand_down_reason}`) — is the shared state four enforcement surfaces read and a small set of CLI verbs mutate. The lifecycle is DETERMINISTIC (engaged by code, not by the model remembering).

**Writers:**

| Transition | Who / how | Guard |
|---|---|---|
| → `active` (engage) | `pretool_skill_gate.record_engagement()` at `PostToolUse(Skill)` when the invoked skill's base name ∈ `RUN_DRIVING_SKILLS` | Requires an explicit payload `cwd`, a genuine non-teammate transcript, and a non-errored `tool_response`. PostToolUse fires only after the tool ran, so a denied/errored Skill call never engages. A resumed session re-invoking the Skill RE-engages the SAME run (identity preserved). |
| keep `phase`/`slug`/`run_id` current | `run_continuity.py --set k=v` (orchestrator, at phase boundaries) | Observability, not enforcement; `--set` refuses `schema`/`status`. |
| → `complete` | `run_continuity.py --mark-complete` (the documented FINAL Phase 8 / B8 / M8 / U9 action, after auto-merge/push) | **Worklist-guarded**: re-runs `pipeline-completion-audit.py --check` and REFUSES if the worklist reports open debt (exit 2) unless `--force`; appends to `run-completion.log`; clears `stop-guard-state.json`. |
| → `stood-down` | `run_continuity.py --stand-down "<user's words>"` | Records the USER's verbatim words to the auditable `pipeline-stand-down.md`; justified ONLY by an explicit user direction. |

**The three skill-set constants (do NOT conflate):** `_PIPELINE_SKILLS` (`pretool_skill_gate.py`, 5 members — the 4 playbooks + `proposal-refiner`) is what SATISFIES gate arm 1; `RUN_DRIVING_SKILLS` (`run_continuity.py`, the 4 playbooks only) is what ENGAGES the marker; `ENGAGEMENT_SKILLS` (`run_continuity.py` = `RUN_DRIVING_SKILLS | {proposal-refiner}`, 5 members) is what counts as OPERATING UNDER the pipeline for the sticky arm / engaged-session detection. `_PIPELINE_SKILLS` and `ENGAGEMENT_SKILLS` have IDENTICAL 5-member membership but are separate constants in separate modules serving separate arms; only `RUN_DRIVING_SKILLS` (the 4-member subset) differs by membership. `proposal-refiner` satisfies arm 1 and marks a session engaged but never STARTS a run (it also runs standalone via `/refine-prompt`), so invoking it never leaves a workspace stuck run-active.

**Readers (who the marker gates):**

1. **Sticky run arm** — `pretool_skill_gate._sticky_run_check()`. While the marker is `active`, a USER-FACING session that has NOT engaged a pipeline skill SINCE ITS LAST COMPACT BOUNDARY is blocked from build/dispatch tools (`_BLOCKED_TOOLS` = Edit / Write / NotebookEdit / Agent / Task*) until it re-invokes the Skill. Stands down for: teammate sessions (`CT6-TEAMMATE` token or a brief-shaped-first-prompt fallback), sidechain / subagent transcripts (no genuine user prompt), already-engaged sessions, a STALE marker (> `CT6_RUN_MARKER_STALE_HOURS`, default 72 h), a workspace paused at `escalation-pending.md`, and any AMBIGUOUS engagement answer on a tail-truncated transcript. Never blocks `Read` / `Bash`.
2. **Continuation guard** — `pipeline-completion-audit.py`. An `active` marker blocks a Stop EVEN WHEN THE WORKLIST IS MOMENTARILY CLEAN (a between-phases run is not done). ENGAGED orchestrator sessions (the marker's `session_id` matches the payload's, OR the transcript shows a pipeline-skill invocation) get BOUNDED NO-PROGRESS PERSISTENCE: the run's progress fingerprint (`run_fingerprint()` — sha256 over `.architect-team/**` stat data + `.git/HEAD`/ref/index + `git status --porcelain`, excluding self-churn files) is compared between stops; a changed fingerprint OR a fresh user-prompt anchor resets the counter — a progressing run is pushed UNBOUNDED (Unbounded solving); an identical fingerprint increments `stop-guard-state.json`'s counter, and at `CT6_MAX_NO_PROGRESS_STOPS` (default 3) the guard writes `escalation-pending.md` and ALLOWS the Stop (a wedged run never infinite-loops). NON-engaged sessions keep the legacy one-nudge semantics + a resume-via-Skill note. `--check` mode deliberately skips the lifecycle guard (Phase 8 runs it while the run is still active).
3. **SessionStart directive** — `sessionstart-run-continuity.py::build_directive()`. On startup/resume/clear/compact, an `active` marker injects "invoke `Skill(<skill>)` FIRST, then resume"; a `compact` source adds a "your context was just compacted" note; a STALE marker downgrades to an informational "clear it" message.

Engagement detection is compact-boundary-aware: `session_engaged_pipeline()` scans the Skill-tool ledger AFTER the last compact boundary (a compacted session must RE-invoke) and reads BOTH the tail-capped slice and a 512 KB HEAD slice (so a long-lived teammate's spawn brief / the original invocation never scrolls out of recognition); ambiguity on a truncated transcript fails open. **Circular-import shape:** `pretool_skill_gate.py` and `run_continuity.py` reference each other (the substrate reuses the gate's transcript record helpers `_read_records` / `_is_user_prompt` / `_skill_invocations` / `_text` / `_timestamp`) via module-object import with lazy attribute access — unavailable ⇒ the arm goes inert (fail open).

**Per-run, not shared (verified in code):** the run-continuity marker and its guard artifacts (`active-run.json`, `stop-guard-state.json`, `pipeline-stand-down.md`, `run-completion.log`, `escalation-pending.md`, `in-progress.md`) are all resolved relative to the WORKING DIRECTORY, NOT via `shared_state_dir()` — the sticky arm and SessionStart use the payload `cwd`, the Stop guard uses `Path.cwd()`, the CLI uses `--root`/cwd. This is correct by design (run-continuity is scoped to "this session's run in this working tree"), but it means a run in an auto-created `architect-team/<slug>` worktree keeps its continuity marker THERE, and a `Stop`/`SessionStart` fired from a different cwd sees no marker (and never gates an unrelated payload — the readers never fall back to the hook process's own cwd). Single-session (non-worktree) users see one merged `.architect-team/` (both roots degenerate to the same path). Everything is gitignored runtime state (see `## Runtime dependencies` for the shared-vs-per-run state model).

### `services/` tier — adapter seams + the separability invariant

`services/` (v3.23.0→v3.28.0, the CT6-6 server tier, program complete) is DESIGN + a runnable stdlib-only core + tests — **NOT a live-deployed service**. The seams ARE the integration surface.

**The shared substrate + 4 services** (`services/separation.py::SEPARATION_MANIFEST`, each marked `separable: True`):

- `common/` — `ed25519.py` (pure-Python RFC-8032 Ed25519, no 3rd-party dep), `handshake.py` (signed submission envelopes + freshness + nonce-replay + a pluggable attestation hook), `bg_runtime.py` (scheduler + per-OS install descriptors + a log-ship interface; GENERATES descriptors, does NOT daemonize), `service_config.py` (the single-Anthropic-key model + the `LLMClient` adapter).

**The 7 injected adapter seams** (`SEPARATION_MANIFEST["seams"]` — the exhaustive list of where a closed/external capability plugs in; `open_stub` = the stdlib fallback the core ships, `closed` = what a real/paid deployment injects):

| Seam | Req | Paid? | Open stub | Closed / injected |
|---|---|---|---|---|
| `attestation_verifier` | SEC-4 | **yes** | `handshake.make_hmac_attestation_verifier` (HMAC) | project-unique genuine-logger proof |
| `entitlements_for` | SMP-4 | **yes** | injected lookup keyed on the VERIFIED public key | entitlement / billing system |
| `LLMClient` | LIB-1 / EVAL-1 / SR-1 | no | `FakeLLMClient` | `anthropic_client()` — real SDK+network, LAZY import inside the fn |
| `Source` | LIB-6 | no | `StaticSource` | `UrlSource` (stdlib `urllib`) / a real scraper |
| `IssueSink` / poster | EVAL-2 | no | `InMemorySink` / `poster=None` | GitHub issues API |
| `transport` | SMP-2 | no | an in-process callable | real HTTP fetch to the project server |
| `bg_runtime` daemon | BG-1…4 | no | generated per-OS install descriptors | an actual OS daemon + off-machine log ship |

**The SEC handshake seam** (concrete): `handshake.make_envelope()` signs `{alg, nonce, ts, payload, attestation}` with the logger's local Ed25519 keypair; `verify_envelope()` checks shape → freshness (±300 s) → nonce-replay → Ed25519 signature → (optional) `attestation_verifier`. The attestation is bound INTO the signed message so it cannot be swapped. The genuine-logger attestation algorithm is the separable closed/paid SEC-4 piece; the open core ships only the pluggable hook + an HMAC stub. Both the triage (EVAL) and seeded-MemPalace (SMP) servers authenticate submissions through this seam.

**The LLM seam** (concrete): `service_config.py` encodes ONE Anthropic key as BOTH the background LLM key and the triage sign-up identity (SEC-2). The real call is `anthropic_client(config)`, which imports `anthropic` LAZILY inside the function body and raises an actionable error if the SDK/key is absent — so no network/SDK dependency leaks into module load. `redacted()` masks the key to its last 4 chars for logs.

**The separability invariant** (`services/separation.py::check_separation()`, REPO-4): parses every `services/**/*.py` with `ast` and asserts each is IMPORT-CLEAN at module load — only stdlib (`sys.stdlib_module_names`, 3.10+) + in-repo modules. The in-repo allow-list is the set of `services/**` module stems PLUS the two cross-tier reuse points imported by bare name (`_EXTRA_INREPO_MODULES = ("phenotypes", "logit")`). Its `_module_load_imports()` walker recurses through every load-time compound statement (`try` / `if` / `with` / `for` / class body) but NOT function bodies — so a hard `try: import chromadb` is a violation, while the lazy `import anthropic` inside `anthropic_client` is allowed. That import-cleanliness IS what makes each service liftable into its own repo; the actual repo SPLIT is a future op — this module DESIGNS + VALIDATES the boundary only. `validate_manifest()` additionally asserts every paid/closed piece names a real seam.

### Cross-session lock layer (v1.0.0)

The user has been asking for cross-session parallelism — *"can I run multiple `/architect-team` invocations at once?"* — since v0.9.x. Given the Agent Teams docs' "one team at a time per Lead" + "no nested teams" constraints, in-session parallelism is impossible; parallelism IS via separate Claude Code sessions. v1.0.0 ships the coordination primitive that makes this safe: **`.architect-team/locks/<scope-hash>.json`**.

**Lock-file shape:**

```jsonc
{
  "holder": "run-<slug>",                   // human-readable run identifier
  "scope_glob": "src/auth/**",              // the file-scope claim
  "acquired_at": "2026-05-28T01:42:00Z",    // ISO 8601 UTC timestamp
  "ttl_seconds": 14400,                      // default 4h (14400s)
  "run_id": "<run-uuid>"                     // unique per /architect-team invocation
}
```

**The four primitives** (all in `hooks/locks.py`, stdlib-only):

| Function | Behavior |
|---|---|
| `acquire_lock(scope_glob, ttl_seconds, run_id)` | Hashes the scope_glob to a filename; if no non-stale lock at that path intersects, writes a new lock file and returns the lock id. If an intersecting non-stale lock exists, returns `blocked: <existing-lock-id>` (the caller's choice whether to queue or surface a conflict). Malformed / missing-field lock files are treated as stale and auto-released. |
| `release_lock(lock_id)` | Removes the lock file. Idempotent — releasing a non-existent lock is a no-op. |
| `detect_stale()` | Returns the lock ids whose `acquired_at + ttl_seconds` is in the past, or whose JSON is malformed / missing required fields. Used by `acquire_lock` internally; can be called standalone for cleanup. |
| `globs_intersect(a, b)` | Returns True when two path-globs could match a common path. Reuses the non-overlapping-file-scope discipline from `team-spawning-and-review-gates`. |

**Use case — two `/architect-team` invocations in separate Claude Code sessions:**

- **Session 1** (Lead A): calls `acquire_lock(scope_glob="src/auth/**", ttl_seconds=14400, run_id="run-A")` → writes the lock at `.architect-team/locks/<hash-of-auth>.json` with `holder: "run-A"`, returns the lock id. Lead A proceeds to dispatch teammates against `src/auth/**`.
- **Session 2** (Lead B, started 10 minutes later): calls `acquire_lock(scope_glob="src/auth/login/**", ...)` → `globs_intersect("src/auth/**", "src/auth/login/**")` returns True → returns `blocked: <Lead-A's-lock-id>`. Lead B's choice: queue (poll until the lock releases or goes stale) or surface a conflict to the user.
- **Disjoint case:** Session 2 calls `acquire_lock(scope_glob="src/billing/**", ...)` → `globs_intersect("src/auth/**", "src/billing/**")` returns False → acquire succeeds, both leads run truly parallel.
- **Crash case:** Lead A crashes; its lock sits at `.architect-team/locks/`. After 4 hours its `acquired_at + ttl_seconds` is in the past; the next intersecting `acquire_lock` invokes `detect_stale()`, removes the stale lock, and succeeds. No human cleanup required.

The lock layer is BEST-EFFORT cross-session coordination, NOT a hard-real-time mutex. If two leads in two sessions both call `acquire_lock` within the same wall-clock millisecond, the filesystem semantics of `os.replace` / atomic write decide the winner; the loser's call detects the winner's lock on its next read and returns `blocked`. The layer guarantees no data corruption and no silent overlap; it does NOT guarantee a strict FIFO acquisition order under perfect concurrency. Acceptable for the user's actual use case (one ad-hoc parallel run per developer per workspace).

This handoff IS the structural primitive that opens cross-session parallelism. Update `hooks/locks.py` in lockstep with the discipline documented in `skills/team-spawning-and-review-gates/SKILL.md`'s non-overlapping-file-scope section.

## Interactive-mockup-discovery flow (v2.1.0)

A new internal contract surface added in v2.1.0. The flow engages when the named oracle artifact at Phase 0.5 is an interactive HTML mockup; otherwise the existing 5 spec_shape categories (component-tree / design-map / api-contract / data-model / hybrid) handle the artifact and v2.1.0 is a no-op.

### Trigger detection (oracle-deriver, Phase 0.5)

`agents/oracle-deriver.md` classifies `spec_shape: interactive-mockup` when ALL of these hold:
- The artifact is an HTML file (or a directory containing `index.html` + assets).
- The artifact carries one or more runtime-interactivity signals: a `<script>` tag, an inline `onclick=` attribute, an `addEventListener` call, OR a `[data-action]` attribute.

When triggered, oracle-deriver dispatches the `interaction-observer` agent BEFORE writing its frozen spec.

### Pass 1 — Observation (interaction-observer)

`agents/interaction-observer.md` (fable, color green) runs in one of two paths:

| Path | When | Mechanism |
|---|---|---|
| Live | Playwright + headless Chrome available | Spin up Playwright, navigate to mockup (single-file HTML served via `python3 -m http.server` on a free port), wait for `document.readyState === 'complete'` + framework ready event, enumerate every interactive element, simulate each interaction with `page.click` / `page.fill` / `page.selectOption` / `page.hover`, capture before/after DOM snapshot, classify the diff into one of the 7 action_kind values |
| Snapshot | Live runtime unavailable (the v2.1.0 plugin self-test path) | Read a pre-captured DOM-interaction-snapshot JSON from the path the orchestrator names in the dispatch brief; iterate through `simulations[]` to produce interactions[] |

Output: a structured `interactions[]` array on the frozen oracle spec. Each entry has shape `{interaction_id, trigger_selector, semantic_label, action_kind, observed_effect, target_url_or_state, evidence_path}` where `action_kind` is the closed 7-value vocabulary `navigate / open-drawer / open-modal / submit / input-text / reveal / no-op`. Per-interaction evidence files (DOM before/after, screenshots, captured fetch URLs) live at `<workspace>/.architect-team/oracle-spec/<change-name>/interaction-evidence/int-NNN.json`.

The observer's Write tool is bounded to the oracle-spec path + the per-interaction evidence files + optionally an observation-log; touching feature source is forbidden.

### Pass 2 — Intent inference (interaction-intuiter, INTENT-INFERENCE mode)

`agents/interaction-intuiter.md` gains a new INTENT-INFERENCE mode that fires when the orchestrator finds the frozen oracle spec carries a non-empty `interactions[]`. The intuiter walks every entry and compares `semantic_label` against `observed_effect` + `target_url_or_state` using the canonical mismatch matrix (10 patterns — Logout / Sign In / Save Draft / Delete / Cancel / Next / Back / Search / Submit / Edit — each mapped to expected intent + mismatch examples). The matrix lives in the intuiter agent body so the rules are auditable from a single source.

For each mismatch, the intuiter emits an `interaction_intent_gap` entry with shape `{gap_id, interaction_id, trigger_selector, semantic_label, observed_action_kind, observed_target, expected_pattern, ambiguity_question, candidate_intents: [...], user_verdict: null, resolved_intent: null}`. Output file: `<workspace>/.architect-team/oracle-spec/<change-name>-intent-gaps.json`.

### Phase −1D bulk-verify integration

The orchestrator folds the intent gaps into the EXISTING Phase −1D bulk-verify gate alongside any per-codebase `INTERACTION_INTUITION_MAP.md` ambiguities. The user sees ONE unified numbered list and resolves all intent-source ambiguities together using the existing `all correct` / `<comma-separated numbers>` / `all incorrect` response format. The drill-down round uses `AskUserQuestion` with each gap's `candidate_intents[]` as options.

After resolution: `user_verdict ∈ {confirmed, corrected, confirmed-stub, deferred}`, `resolved_intent` is set to the canonical `action_kind:target` (e.g., `"navigate:/sign-in"`). The orchestrator writes `resolved_intent` BACK to the corresponding interactions[] entry on the frozen oracle spec at `<workspace>/.architect-team/oracle-spec/<change-name>.json`, then flips `_human_review_required: false`.

### Layer 3 verification (verify_interactions_honored)

At Phase 3 review-evidence write time (when an in-scope task touches a component referenced by an interactions[] entry), the orchestrator invokes `hooks/vao_tools.py::verify_interactions_honored(built_components, oracle_spec)` (or the CLI `python3 hooks/vao_tools.py verify-interactions-honored --components C --oracle O --out OUT`). The tool walks every interactions[] entry, prefers `resolved_intent` over the observed action when present, asserts the built code's handler matches with three named severities:

| Severity | When |
|---|---|
| `missing-handler` | Oracle says this trigger has an effect; built code has no handler at all |
| `intent-violated` | Resolved_intent says X; built code does Y with the same action_kind but a different target (the canonical mockup-lies case) |
| `action-kind-mismatch` | Action_kind itself differs (oracle says open-modal; built code navigates) |

Output verdict JSON at `<cwd>/.architect-team/vao-verdicts/<task-id>-interactions-honored.json` with shape `{tool, matched, gaps, honored_count, total_count, verdict_at}`. Sorted-keys + indent=2 (deterministic / bit-stable — same contract as the other 5 Layer-3 tools).

### Schema v7 citation contract

`hooks/review_evidence_schema.py` schema v7 includes an OPTIONAL `interactions_honored_review` field. Required ONLY when the run's oracle spec carries a non-empty `interactions[]` array; n/a in all other cases. Accepts:

- String shape: `"pass"` / `"n/a"` / `"fail"` (legacy review form).
- Dict shape: `{verdict: "pass", verdict_path: ".architect-team/vao-verdicts/<task-id>-interactions-honored.json"}` (canonical v7 citation form — the cited verdict file is the source of truth at hook level; an agent's prose `verdict: pass` that cites a verdict JSON with non-empty `gaps[]` is hook-blocked).

`REQUIRED_EVIDENCE_FIELDS` stays at 17. The new field lives in the new `OPTIONAL_VAO_FIELDS` tuple. `VALID_INTERACTIONS_HONORED_VALUES = {"pass", "n/a", "fail"}`. v2.0.0 evidence files (which lack the field entirely) continue to validate — the field's optional-ness is the v2.1.0 backwards-compat guarantee.

v3.14.0 adds the THIRD optional field on the same contract: `appearance_scope_review` — required only when the slice's diff touches frontend presentation surface (styling files, components, templates, routes, assets); same string-or-dict shapes; `VALID_APPEARANCE_SCOPE_VALUES = {"pass", "n/a", "fail"}`; the hook BLOCKS `fail` (an appearance-affecting delta with no mandate trace under the `strict` / `propose` appearance modes); `n/a` requires `appearance_scope_review_note`. Pre-v3.14.0 evidence files continue to validate. Canonical rules: `common-pipeline-conventions` `## Appearance-change policy discipline (v3.14.0)`.

- No new HTTP / DB / queue / file-system surface.
- No new env var, no new config file, no new opt-in.

The flow extends the v2.0.0 VAO framework's Layer 1 (oracle-deriver) and Layer 3 (vao_tools) additively — both the new `spec_shape: interactive-mockup` value and the new `verify_interactions_honored` tool are pure additions; the existing 5 spec_shape categories and 5 Layer-3 tools are unchanged.

## Runtime dependencies (v1.1.0)

### Teams-mode dependencies (v1.0.0)

v1.0.0 adds a NEW DEPENDENCY for teams-mode dispatch: **`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`** — set in the shell environment OR as `{"env": {"CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"}}` in `~/.claude/settings.json` — AND **Claude Code ≥ 2.1.32**. Both are required for teams mode; without either the pipeline silently falls back to subagents mode (the v0.10.0 behavior, unchanged). The mode-detection helper `scripts/setup/teams_mode.py::is_teams_mode_available()` makes the decision once at pipeline startup; the decision is recorded in `intake-state.json` and the pipeline never re-checks. `/architect-team:architect-team-setup` (`scripts/setup/setup.py`) checks both at install time and offers to add the env var to `~/.claude/settings.json` with explicit user consent. The `--no-teams` flag on any of the three pipeline commands (`/architect-team`, `/architect-team:bug-fix`, `/architect-team:mini`) forces subagents mode even when teams are available.

### Worktree-aware state resolution (v1.1.0)

v1.1.0 introduces a new INTERNAL runtime dependency (not user-facing — no env var, no config, no opt-in): **the lock layer and the MemPalace integration both resolve their root path via `scripts/setup/worktree_paths.py::shared_state_dir()`**, which routes through `git rev-parse --git-common-dir` to identify the MAIN worktree's `.architect-team/` directory. This closes the v1.0.0 gap where two `/architect-team` sessions in two git worktrees of the same repo each got their own isolated locks dir + their own isolated MemPalace, defeating the cross-session coordination intent.

**Resolution semantics** (canonical, applied by every consumer):

| Function | Returns | Used for |
|---|---|---|
| `shared_state_dir() -> Path` | parent(`git rev-parse --git-common-dir`) / `.architect-team` (or cwd / `.architect-team` if git probe fails) | `locks/` (via `hooks/locks.py` default), `.mempalace/palace/` (via `mempalace-integration` skill), `run-history/` |
| `run_state_dir() -> Path` | `Path.cwd() / '.architect-team'` (always per-worktree) | `reviews/`, `teammates/`, `handoffs/`, this-run's `openspec/changes/<slug>/`, this-run's findings + refined-prompts, `vao-verdicts/`, `inbox/`, and (v3.30.0) the run-continuity markers `active-run.json` / `stop-guard-state.json` / `escalation-pending.md` / `in-progress.md` / `pipeline-stand-down.md` / `run-completion.log` (resolved against the session cwd, never via `shared_state_dir()`) |
| `is_worktree() -> bool` | True iff `git rev-parse --git-dir` != `git rev-parse --git-common-dir` (absolute-path comparison) | downstream callers that want to know |

**The 3-layer model** the helper enables (canonical, applied to cross-session coordination overall):

| Layer | Concern | Mechanism | Resolution |
|---|---|---|---|
| Filesystem isolation | Two sessions can't clobber each other's working tree, index, or branch | Git worktrees (`git worktree add`) | Each session checks out its own worktree on its own branch — use `superpowers:using-git-worktrees` for lifecycle |
| Architectural coordination | Two sessions can't both decide to refactor the same scope-glob | `.architect-team/locks/` JSON locks (v1.0.0) | `shared_state_dir() / 'locks'` — both worktrees see the same lock dir |
| Context sharing | Session B sees what session A produced | MemPalace + CLAUDE.md + codebase maps | `shared_state_dir() / '.mempalace' / 'palace'` — both sessions share one palace |

**Backwards compatibility:** single-session users (no worktrees) see ZERO behavior change. `shared_state_dir()` and `run_state_dir()` degenerate to the same path in a non-worktree clone. The lock layer reads/writes the same location it did in v1.0.0. No env var, no flag, no opt-in. The lock layer's explicit `locks_dir=` parameter (used by `tests/test_locks.py` for test isolation) is preserved verbatim — all 17 v1.0.0 lock tests pass unchanged after the v1.1.0 default-resolution change.

**Failure modes:** every git subprocess invocation in `worktree_paths.py` is wrapped to fall back to `Path.cwd() / '.architect-team'` on any failure (not a git repo, git missing, broken metadata, empty stdout). `hooks/locks.py`'s `_default_locks_dir()` further wraps the helper import in a try/except — if `worktree_paths.py` is unreachable for any reason, the lock layer falls back to the legacy `DEFAULT_LOCKS_DIR` constant, so the public lock API never raises. The cross-worktree coordination guarantee degrades in these edge cases (each worktree's lock layer becomes isolated again), but no run is interrupted.

**Out of scope for v1.1.0:** cross-CLONE coordination (two separate clones of the same project on disk do NOT coordinate via this layer — the lock files + MemPalace are repo-local, not machine-wide); worktree lifecycle automation from inside the pipeline (the `superpowers:using-git-worktrees` skill already handles this).

### Auto-worktree lifecycle (v1.2.0)

v1.2.0 brings worktree-lifecycle automation IN-scope for the pipeline. v1.1.0 made state-resolution worktree-aware but left worktree CREATION as a manual user step (the user ran `git worktree add` before invoking `/architect-team`). v1.2.0 closes that gap: every invocation of the three pipeline-driving slash commands — `/architect-team`, `/architect-team:bug-fix`, `/architect-team:mini` — automatically creates a fresh worktree at `<parent-of-repo>/<repo-name>-<slug>/` on branch `architect-team/<slug>` BEFORE invoking the pipeline skill. The user's main checkout stays on whatever branch they were on; the run is filesystem-isolated; concurrent runs work without any manual setup. The 7 utility commands (`/architect-team:visual-qa`, `/architect-team:editability-audit`, `/architect-team:refine-prompt`, `/architect-team:memory`, `/architect-team:mempalace-install`, `/architect-team:architect-team-setup`, `/architect-team:mini-review-sweep`) do NOT get the step — they operate against the current checkout because their work is inspection / configuration / replay, not feature delivery.

**Helper:** `scripts/setup/worktree_lifecycle.py` — sibling to v1.1.0's `worktree_paths.py`. Stdlib-only. Four public functions:

| Function | Purpose | Failure mode |
|---|---|---|
| `create_run_worktree(slug, base_branch="main", parent_dir=None) -> Path` | `git worktree add -b architect-team/<slug> <parent>/<repo>-<slug>/ <base>`. Collision handling appends `-2`, `-3`, ... when path or branch is taken (bounded at 999 attempts). | Raises `RuntimeError` with an actionable message on parent-dir-not-writable, base-branch-missing, `git worktree add` failure, or slug exhaustion. The slash command surfaces the error verbatim and stops — no silent fallback. |
| `cleanup_run_worktree(worktree_path, remove_branch=False) -> None` | `git worktree remove <path>` + optional `git branch -d architect-team/<slug>`. | Idempotent on already-gone worktree (catches `is not a working tree` / `is not a registered` / `no such file or directory`). Falls back to `--force` removal once before raising. `not fully merged` branch deletion raises with a manual-cleanup hint. |
| `current_worktree_is_run() -> bool` | True iff `git rev-parse --abbrev-ref HEAD` starts with `architect-team/`. | Returns False on any subprocess failure (not in a git repo, git missing, detached HEAD). Never raises. |
| `current_run_slug() -> str | None` | Extracts `<slug>` from `architect-team/<slug>`. | Returns None on non-run branches, detached HEAD, or subprocess failure. |

**Slash-command integration:** each pipeline-driving slash command body has a `## Auto-worktree creation (v1.2.0)` section between argument parsing (+ refinement) and skill invocation. The step is skipped when ANY of these holds: `--no-worktree` flag (or natural-language opt-out — *"no worktree"*, *"don't create a worktree"*, *"single tree"*, *"in place"*, *"in current tree"*); OR re-entry from inside an existing `architect-team/*` worktree (detected via `current_worktree_is_run()`). Otherwise the step derives a slug, invokes the helper via the polyglot `python3 -c '...' || python -c '...'` pattern, chdirs into the new worktree, and emits a one-line note to the user before invoking the pipeline skill.

**Cleanup semantics:** NOT automatic. The pipeline emits a recommendation at Phase 8 / B8 / M7 success — *"Your run worktree is at `<path>` on branch `architect-team/<slug>`. To clean up: `git worktree remove <path> && git branch -d architect-team/<slug>`."* — and the user decides when to clean up. The user may want to inspect the run, layer a follow-up via re-entry detection, or run manual tests against the run-branch tip.

**Backwards compatibility:** v1.1.0 single-tree users pass `--no-worktree` on every invocation for verbatim v1.1.0 behavior. The auto-worktree step adds ~1s of pipeline startup time (the `git worktree add` subprocess); negligible for a feature-shipping pipeline. Re-entry detection means existing in-flight runs already on `architect-team/<slug>` branches are picked up correctly — re-invoking `/architect-team` from inside one of those does NOT create a nested worktree.

**Out of scope for v1.2.0:** auto-cleanup after Phase 8 merge (pipeline reports + recommends; user decides — future v1.3+ could add `--auto-cleanup`); multi-worktree dispatch (each pipeline run gets ONE worktree; teammates dispatch INTO that worktree, not into per-teammate worktrees — v1.3+ idea); auto-worktree for the 7 utility commands (they're inspection / configuration / replay, not feature delivery — limit auto-worktree to the 3 pipeline-driving commands).

## Code & Data Lineage Graph (CDLG) integration (v3.8.0)

The v3.8.0 lineage foundation adds INTERNAL integration surfaces — no new external dependency, no new HTTP / DB / queue / 3rd-party API. The CDLG is an in-process, on-disk artifact (a graph JSON) produced by an agent and consumed by the pipeline's own diagnosis + parallel-dispatch machinery. The Playwright / runtime-execution dependency it grounds against (`code-path-witness.json`) is ALREADY a plugin artifact (v0.9.31 / v0.9.32) — v3.8.0 reuses it, never re-implements execution capture.

### CDLG ↔ runtime-witness reconciliation (the trust gate)

The load-bearing contract. `hooks/lineage_graph.py::reconcile_with_witness(graph, witness)` cross-checks every asserted CDLG edge against the existing **`code-path-witness.json`** runtime-execution witness:

- **Producer:** the `endpoint-tracer` agent (via the `endpoint-trace-mapping` skill) emits a candidate CDLG from live polyglot extraction — a graph of `func://` / `endpoint://` / `asset://` nodes and `calls` / `reads` / `writes` / `modifies` / `serves` / `originates` / `serves_route` edges.
- **Ground truth:** `code-path-witness.json` (the v0.9.31 / v0.9.32 wrong-code-path execution witness — already captured during Playwright / backend test runs).
- **Reconciliation output:** edge **recall** (witnessed edges present in the CDLG) + **hallucination rate** (CDLG edges asserting execution that the witness never observed) + the `witness_gate` verdict. An edge whose execution is unwitnessed is downgraded, not trusted — so the CDLG that downstream phases consume is witness-verified, not LLM-asserted-only.
- **Honest boundary:** the deterministic reconciliation + recall/hallucination math is real and unit-tested (`tests/test_lineage_graph.py`); the live polyglot extraction that *produces* the candidate graph against an arbitrary target repo is the agent's runtime job and is **kill-gated / NOT claimed proven** (per `docs/LINEAGE_UPGRADE_REQUIREMENTS.md` §7.1).

### CDLG ↔ pipeline diagnosis (P2 consumer rewire)

`diagnostic-research-team` (the Phase-2 pre-fix diagnostic stage) now **consumes the witness-verified CDLG** instead of re-tracing the code path from scratch. The CDLG is the call-graph the three `diagnostic-researcher` agents reason over; the `bug-fix-pipeline` P0 isolation reorder (replicate → scope-isolate → EXECUTED light FE/API discriminant → call-map → diagnose) front-loads the cheap evidence-backed `endpoint-trace-mapping` call-map BEFORE the expensive deep diagnosis. The `fe_api_verdict` discriminant + the `layer_fixed` reality are captured by `hooks/run_metrics.py` so wrong-layer fixes are measurable.

### CDLG ↔ parallel-execution locks (`cdlg_overlap`, P4)

`hooks/locks.py::cdlg_overlap(...)` extends the v1.0.0 cross-session lock layer from FILE-scope intersection (`globs_intersect`) to **call-graph-closure intersection**. Two slices whose edited file sets are disjoint can still collide if both call-closures reach a shared HOT CALLEE (a function reachable from both); `cdlg_overlap` flags that case so the Lead can serialize the colliding work rather than dispatch it truly-parallel. The lock layer's existing `acquire_lock` / `release_lock` / `detect_stale` semantics are unchanged; `cdlg_overlap` is an additional, graph-aware overlap predicate consulted at dispatch time.

### CDLG ↔ MemPalace function-level records (P5)

`mempalace-integration` adds **function-level lineage records keyed by `func://`** — the rename-stable content-fingerprint ID nomenclature from `hooks/lineage_graph.py` is the join key that lets MemPalace dedup + diff lineage facts across runs even when a function is renamed or moved. No new external integration: MemPalace remains the optional v0.9.4 local-first store; v3.8.0 just adds a new record kind keyed by the new ID scheme.

### Run-metric instrumentation (P0.5)

`hooks/run_metrics.py` writes a per-run metrics JSON at `<workspace>/.architect-team/run-metrics/<run_id>.json` (the §6 success metrics: `dev_loop_iterations`, first-pass-fix, oscillation / bug-still-present / fix-regression counts, the FE/API discriminant verdict vs the layer actually fixed, CDLG recall / hallucination) and mirrors it to MemPalace run-history. It is the measurement surface for the frozen-bug-benchmark protocol — purely internal, on-disk, no external call.

## Deployment Topology

The architect-team plugin deploys as a Claude Code plugin via:

1. User runs `/plugin marketplace add https://github.com/paulingram/claude-skills.git`. The harness git-clones to `~/.claude/plugins/marketplaces/architect-team-marketplace/`.
2. User runs `/plugin install architect-team@architect-team-marketplace`. The harness copies the plugin into `~/.claude/plugins/cache/architect-team-marketplace/architect-team/<version>/` and registers it in `~/.claude/plugins/installed_plugins.json`.
3. User runs `/reload-plugins`. The harness reads `.claude-plugin/plugin.json`, `hooks/hooks.json`, every `skills/<name>/SKILL.md`, every `agents/<name>.md`, and every `commands/<name>.md` and registers them as namespaced (`architect-team:*`).
4. User runs `/architect-team:architect-team-setup` once to install external deps (openspec / pytest / playwright / chromium).
5. User invokes `/architect-team:architect-team <path>` to run the pipeline.

There is no server-side deployment. All execution is local in the Claude Code session.

## Failure Modes (cross-system)

- **Missing prerequisite plugin (superpowers / cartographer / ralph-loop)**: setup script reports `[missing]` with the install command. Orchestrator skill bodies reference these by name and would fail at first invocation if absent.
- **Stale marketplace clone**: pre-v0.2.2 manual `git pull` was needed because `/plugin marketplace add` did not re-fetch on remove+re-add. Post-v0.2.2 confirmed: `/plugin marketplace update <name>` does fetch and reports "1 plugin bumped" when a new commit is on origin.
- **Hook over-aggressive**: pre-v0.2.2 the review-gate hook blocked every TaskUpdate. v0.2.2 scoped it to teammate tasks only.
- **`uv pip install` without venv**: pre-v0.1.1, setup script failed to install Playwright on machines with `uv` on PATH. v0.1.1 detects venv and adds `--system` when none active.
- **Skill name collision with command**: pre-v0.2.0, the skill named `architect-team` collided with the slash command of the same name; Skill tool resolved to the command body. v0.2.0 renamed skill to `architect-team-pipeline`.
- **`disable-model-invocation: true` on a skill the command needs to invoke**: pre-v0.2.1, the orchestrator skill had this flag set, blocking the Skill tool. v0.2.1 removed it.

## Architectural notes & inherent limits

The v0.2.3-era open questions (REQ-001 `$ARGUMENTS` propagation, REQ-002 path-traversal sanitization, REQ-003 test-coverage gaps, REQ-004 hook-rejection escalation policy, REQ-005 spec drift) are all **resolved** — see `CHANGELOG.md`. The v0.9.9 logic-implementation review surfaced and closed three tiers of holes. What remains are the inherent limits of an LLM-agent architecture — recorded here honestly rather than papered over:

- **The orchestrator cannot be hooked mid-run.** It is the main session; no hook gates its mid-run behaviour. Every phase discipline is trust-based Markdown. The mitigations are real but bounded: the `Stop` hook (`pipeline-completion-audit.py`) gates the orchestrator's *terminal* state, the Phase 8 `--check` gate makes "clean pass" a checked fact, and the v3.30.0 run-continuity continuation guard now also blocks an ARBITRARY mid-run Stop while the `active-run.json` marker is unresolved (with bounded auto-escalation on a wedged run, so it never infinite-loops) — but none of these can enforce "did you actually run Phase 5's reviews" mid-run; they gate the edges (terminal state, tool dispatch, session start), not the interior.
- **The automated test suite is structural, not behavioural.** The 5465 pytest tests (+ 4 skipped, across 199 test files — canonical Windows-with-PyYAML; 5469 collected, 5466 + 3 on POSIX-without-PyYAML per CODEBASE_MAP's Tests-section environment note) prove the plugin's Markdown / JSON / hook scripts are well-formed and that components do not drift apart (`test_cross_consistency.py`) — and, for the notifier, that its config/provider/dispatch/CLI logic behaves correctly with the genuinely-external transports (`smtplib.SMTP`, `urllib.request.urlopen`) mocked. They do NOT prove the live multi-agent pipeline produces correct software — behavioural / integration testing of the orchestration is outside an automated suite by nature.
- **Runs are non-deterministic.** Re-running the pipeline on the same input yields different teammates, classifications, and convergence. The evidence-artifact trail + the Stop-hook audit make outcomes *auditable*, not *reproducible*.
- **Single-codebase 3-explorer convergence is degenerate** but the skill still mandates running it (it must produce the INTEGRATION_MAP.md that `reuse-first-design` consults). Accepted as-is.
