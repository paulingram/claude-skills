# architect-team plugin

## Codebase Overview

This repo IS the source of the `architect-team` Claude Code plugin — a spec-to-production multi-agent coding pipeline. As of **v2.1.0** it ships **29 skills, 30 named agents, 13 slash commands**. **v2.1.0 adds the Interactive-Mockup Discovery framework** — orthogonal extension to the v2.0.0 VAO oracle-deriver (Layer 1) closing two gaps when the oracle artifact is an interactive HTML mockup (Claude Code's artifact-style mockups). Pass 1 (the new `agents/interaction-observer.md`, opus, color green) runs the mockup in headless Chrome via Playwright (or reads a pre-captured DOM-interaction snapshot for stdlib-only tests), enumerates every interactive element, simulates each interaction, and records the observed effect into a structured `interactions[]` array on the frozen oracle spec. Each entry: `{interaction_id, trigger_selector, semantic_label, action_kind, observed_effect, target_url_or_state}` where `action_kind` is the closed 7-value vocabulary `navigate / open-drawer / open-modal / submit / input-text / reveal / no-op`. Pass 2 (extension to `agents/interaction-intuiter.md` — new INTENT-INFERENCE mode) compares every entry's semantic_label against its observed_effect using the canonical mismatch matrix (10 semantic patterns — Logout/Sign In/Save/Cancel/Delete/Next/Back/Search/Submit/Edit) and emits `interaction_intent_gap` entries for any mismatch ("Logout button routes to /dashboard" → flagged). Gaps flow through the EXISTING Phase −1D bulk-verify gate; the user-resolved canonical intent ("Logout SHOULD route to /sign-in") — not the mockup's broken literal behavior — becomes the binding contract. Layer 3 gains a 6th tool `verify-interactions-honored` (deterministic, sorted-keys + indent=2, stdlib-only) that walks every interactions[] entry, prefers `resolved_intent` over the observed action when present, and asserts the built code's handler matches with three named severities: `missing-handler` / `intent-violated` / `action-kind-mismatch`. Schema v7 gains an OPTIONAL `interactions_honored_review` field (REQUIRED only when the run's oracle spec carries non-empty interactions[]; n/a otherwise) — v2.0.0 evidence files continue to validate. Canonical synthetic fixture `tests/fixtures/vao/interactive-mockup-logout-misroute.json` reproduces the "mockup lies" pattern. 63 new tests (`tests/test_vao_interactions_honored.py` 38 + `tests/test_interactive_mockup_discovery.py` 23 + 2 README/registration consistency). Backwards-compatible additive release; the new skill, agent, tool, schema field, and agent extensions are all purely additive. Live headless Chrome wiring deferred to v2.1.x (the agent contract is documented; the observer reads pre-captured snapshots for now). **v2.0.0 shipped the Verified Agent Output (VAO) framework** — six layers of machine-mediated verification of agent claims closing the class of "agent silently does the wrong thing" failures the plugin had been patching one at a time (v1.4 scope, v1.6 git, v1.7 frontend-fake-data, v1.8 agent-resume, heirship 1-4). **Layer 1** the new `agents/oracle-deriver.md` (opus, read-only, color: cyan) dispatched at the new Phase 0.5 between Phase 0 (Detection & Normalization) and Phase 1 (Planning Validation) whenever the requirement contains a parity verb (match / rebuild / mirror / parity / make like / replicate) OR names an oracle codebase / design mockup / reference URL — walks the named oracle deterministically and produces a frozen structured spec at `<workspace>/.architect-team/oracle-spec/<change-name>.json` with five possible spec_shape values (component-tree / design-map / api-contract / data-model / hybrid); user-confirmation gate flips `_human_review_required` false; downstream layers measure against it. **Layer 2** the new `agents/adversarial-reviewer.md` (opus, read-only, color: orange) dispatched alongside every Phase 3 teammate with one of 5 role-paired shapes (oracle-divergence-hunter for parity-verb / fake-data-hunter for backend-dep / git-discipline-hunter for shared-tree always-on / hardcoded-literal-hunter for dynamic-value / general-anti-pattern-hunter for default) — writes the adversarial_review block into the SAME shared evidence file; schema v7 requires BOTH the existing independent_review AND adversarial_review to pass. **Layer 3** `hooks/vao_tools.py` (stdlib-only) ships 5 deterministic verification tools: verify-oracle-match (structural diff with sorted-keys + indent=2 for bit-stable output), verify-baseline-clean (bash-history audit for the 6 v1.6.0-forbidden git ops — stash / reset --hard / rebase / commit --amend / checkout other-branch / clean -f — while letting through legit read ops), verify-no-fake-data (placeholder names + MSW handlers + page.route fulfill stubs + lorem ipsum + hardcoded money + oracle-declared dynamic-value literals; skips test files), verify-every-element (coverage check + non-stub handlers + Playwright tests), and the heirship-amendment verify-rendered-parity (operates on RENDERED DOM + screenshot pixel-diff, NOT source — catches the canonical case where `<TaCrumbs />` exists in both candidate and oracle source but mounts at different rendered parent paths). **Layer 5** structural test enforcement — 154 new VAO tests across `tests/test_vao_tools.py` (32 — positive/negative/determinism for the 5 tools), `tests/test_vao_skill_invocation_audit.py` (55 — slash + prose regex coverage + audit semantics + docs assertions), `tests/test_verified_agent_output.py` (48 — canonical skill body + schema v7 + openspec assertions), `tests/test_vao_fixtures.py` (19 — round-trip every canonical fixture). Plus 7 canonical synthetic fixtures under `tests/fixtures/vao/`: scope-narrowing.json, git-stash-clobber.json, frontend-fake-data.json, oracle-structure-mismatch.json, chrome-mount-level-mismatch.json (THE canonical heirship fixture — same element in both source trees, different rendered mount levels), execution-time-variance.json (the heirship "addressed with residual variance" pattern: agent's inline verdict says pass but the cited verify-rendered-parity verdict path JSON shows matched: false), skill-not-invoked.json (verbatim heirship transcript: user typed /architect-team:architect-team but the ledger contains ZERO Skill invocations). **Layer 6** (ALWAYS-ON — cannot be disabled via --no-vao; opting out IS the failure mode it catches) the new `hooks/skill_invocation_audit.py` (stdlib-only Stop-hook auditor, 313 lines) closes the heirship "applied methodology by hand" failure: parses the session transcript for explicit user Skill-invocation requests in two surface forms (slash-command for 13 user-invocable commands + prose form with 6 verbs `use`/`using`/`invoke`/`run`/`fire`/`with` + optional `the` + optional `/` + command name); cross-checks against the session tool-call ledger at `.architect-team/run-history/<run-id>-toolcalls.jsonl`; exits 2 on any unmatched request with the canonical failure report; writes verdict JSON to `<workspace>/.architect-team/vao-verdicts/<run-id>-skill-invocation-audit.json`. The user-precedence rule (canonical home: `common-pipeline-conventions/SKILL.md` `## Skill-invocation discipline (v2.0.0)` — new section) — user explicit instructions OVERRIDE the "skill already invoked, do not re-execute" system note; applying methodology by hand is FORBIDDEN because it bypasses every VAO layer. Schema v6 → v7 BREAKING change: 5 new required evidence fields citing the on-disk tool verdict — oracle_match_review, baseline_clean_review, no_fake_data_review, adversarial_review, skill_invocation_audit. Each accepts either string (pass / n/a / fail — legacy form) OR dict {verdict, verdict_path} (canonical citation form); the legacy visual_fidelity_review also gained dict-shape support. The hook BLOCKS any evidence file missing any field or carrying a fail verdict. **Layer 4** (run-history shape detection — vao detect-shape tool reading `.architect-team/run-history/`) and the per-pipeline Phase 0.5 inline-dispatch wiring in the 3 pipeline SKILL.md bodies are DEFERRED to v2.1.x — the agents exist, the canonical contract is documented in `skills/verified-agent-output/SKILL.md`, and the orchestrator can invoke per the skill body. The original v1.8.0 reliability piece (`scripts/setup/agent_resume.py` — `is_truncated` / `wrap_agent_result` / `read_checkpoint`) remains intact under v2.0.0: — a reliability gap distinct from v2.0.0's verified-agent-output framework. A real-world background `dv-attorney` agent ran 68 tool-calls of real work; the final report message was lost to a harness-level stream timeout (rate-limit cutoff); the orchestrator saw an empty result and treated the agent as failed; the work was on disk the whole time; the user had to manually `redispatch and continue` so the agent could re-emit its verdict from already-loaded context. v1.8.0 automates that recovery and adds a checkpoint discipline so the resumed agent doesn't re-do the 68 tool calls. Four enforcement points (same layered pattern as v1.6.0/v1.7.0): (1) the new `scripts/setup/agent_resume.py` helper (`is_truncated` detects truncation via 3 heuristics — empty/sub-50-char output, rate-limit / stream-timeout markers, missing `Status:`/`DONE`/`BLOCKED`/`NEEDS_CONTEXT` report-format markers; `wrap_agent_result` invokes a dependency-injected `SendMessage` on truncation, merges resumed output with original, caps at 2 attempts, surfaces `resumed_failed=True` on exhaustion without raising; `read_checkpoint` reads `.architect-team/agent-checkpoints/<agent-id>.json` with default-dir via the v1.1.0 `shared_state_dir()` lazy import); (2) two new canonical sections in `skills/common-pipeline-conventions/SKILL.md` — `## Background-agent resume discipline` (orchestrator MUST wrap every dispatch result; documents the wrap-call rule + 3 truncation heuristics + 2-attempt cap + user-surfacing on `resumed_failed`) and `## Agent checkpoint discipline` (long-running agents (>20 tool calls expected) write checkpoints every ~10 calls; documents the path + schema + cadence + resume-reads-checkpoint pattern); (3) one-paragraph reference in each of the 3 pipeline SKILL.md bodies enumerating the dispatch phases; (4) uniform `## Checkpoint discipline` section in all 27 `agents/*.md` files (inserted AFTER `## Forbidden git operations` for stable cross-agent location). 42 new tests in `tests/test_agent_resume_discipline.py` audit the structural discipline + the helper behavior. Backwards-compatible — purely additive; well-behaved runs that don't hit stream timeouts see no behavior change. **Orthogonal to v2.0.0** — VAO on the separate branch is unaffected; the v1.8.0 helper layers cleanly underneath if v2.0.0 is later approved. **v1.7.0 added the Frontend Missing-API Discipline** — orthogonal to v1.6.0's teammate-git-discipline: when a frontend agent encounters a UI element that needs a backend API which does NOT yet exist, the previous discipline didn't tell the agent what to do — and the predictable failure modes were the four downstream defects each existing gate catches AFTER the round trip is wasted (fake the data → caught by `dynamic-value-discovery`; mock the endpoint → caught by `playwright-user-flows`; hardcode the response → caught by `dynamic-value-discovery`; silently stub the UI → caught by `interaction-completeness`). v1.7.0 ships the explicit alternative at four enforcement points: (1) the new `## Frontend missing-API discipline` section in `common-pipeline-conventions/SKILL.md` (canonical home — names the 4 anti-patterns + the right pattern: write an SR with `origin.kind: "missing-api-for-frontend-element"`, pause that element's work, continue on the rest of the slice, return to wire when the orchestrator re-dispatches with the SR resolved); (2) per-agent `## Missing-API discipline` section in `agents/frontend.md` (authoring side, with a worked example for a `<UserAvatar>` component needing `GET /api/users/me`) + `## Missing-API SR intake` section in `agents/backend.md` (resolver side, surfacing the actual endpoint shape in the dispatch report so the frontend can confirm before wiring); (3) `agents/system-architect.md` Phase 2 architect brief — new ordering-dependency check for every `both`-layer requirement (decide between sequencing backend-first or authorizing the frontend to surface missing-API SRs, the default); (4) the new `pending-backend` element classification in `skills/interaction-completeness/SKILL.md` (the 5th classification — UI exists, awaiting an SR-tracked endpoint; SR-linkage rule: reviewer accepts `pending-backend` only with the matching open SR) + the new `missing-api-for-frontend-element` SR origin-kind in `skills/team-spawning-and-review-gates/SKILL.md` with documented routing (backend dispatched FIRST, NOT through `diagnostic-research-team` — this is a known-shape backend requirement, not a test failure; frontend re-dispatched on backend completion). 26 new tests in `tests/test_frontend_missing_api_discipline.py` audit the structural discipline. Backwards-compatible — purely additive discipline; no runtime detector. **v1.6.0 added the Teammate Git Discipline** — a real-world failure surfaced in a separate user session exposed a discipline gap: four teammates dispatched in parallel against the same working tree each ran `git stash` to verify their work against baseline; the concurrent stash + pop operations interleaved catastrophically, the reflog showed 10+ consecutive `reset: moving to HEAD` entries, and three of four teammates' work was lost (only the last writer survived). The plugin had no rule forbidding teammates from running destructive git operations, so the teammates did. v1.6.0 ships the discipline at four enforcement points (same shape as v1.4.0 scope-discipline): (1) the new `## Teammate git discipline` section in `common-pipeline-conventions/SKILL.md` is the canonical home — names the 6 forbidden destructive operations (`git stash` / `git stash pop`, `git reset --hard`, `git rebase`, `git commit --amend`, `git checkout <other-branch>` / `git checkout .`, `git clean -f`), documents the heirship-app-v2 worked example with the smoking-gun reflog signature (`reset: moving to HEAD` repeated 10×), and names the right pattern. (2) The 3 pipeline bodies each carry a v1.6.0 anti-pattern entry in `## Operating rules (non-negotiable)`. (3) All 27 `agents/*.md` files gain a `## Forbidden git operations` section as a uniform 5-line block between `## Operating context` and the next H2. (4) The `team-spawning-and-review-gates/SKILL.md` documents the `## Baseline SHA capture` orchestrator-side mechanics — the orchestrator runs `BASELINE_SHA=$(git rev-parse HEAD)` at run start, persists it to `intake-state.json` as `baseline_sha`, and carries it in every teammate's spawn brief (extending the v0.9.13 manifest schema); teammates use `git diff $BASELINE_SHA -- <my-files>` for baseline verification instead of `git stash`. 265 new tests in `tests/test_teammate_git_discipline.py` audit the structural discipline. **v1.5.0 added Dispatch-Mode Observability** — the user's direct question *"how do I know if a team is deployed via agent teams vs subagents, can we show an indicator"* exposed a real observability gap (v1.0.0 made the decision silent: it landed in `intake-state.json` but no user-visible signal surfaced). v1.5.0 ships three pieces: (1) the new `format_dispatch_banner()` helper renders a one-block banner naming the active mode + (in the fallback case) the diagnosed reason; (2) all 3 pipeline-driving slash commands now print the banner as their FIRST user-visible action (before the v1.3.0 auto-cleanup, before argument parsing); (3) the new `/architect-team:status` command (13th) reports dispatch mode + active worktrees + open SRs + last completed run; (4) the Phase 8 / B8 / M7 commit-message templates gain a `Dispatch-Mode: <teams|subagents>` trailer so `git log --format=%(trailers)` can answer "which mode produced this commit?" archeologically. The banner is informational, never gating — subprocess failure surfaces a one-line note and the run continues. The dispatch decision itself is unchanged from v1.0.0. **v1.4.0 added the Scope Discipline** — a user-reported plugin-level gap: agents using the package were silently NARROWING the user's prompt at intake (e.g., *"match the oracle"* read as *"enrichment + hardcoded data purge"* with the visual rebuild deferred). The v0.9.36 anti-deferral discipline forbade the MID-RUN version; v1.4.0 extends the forbiddance to INTAKE. The new `## Scope discipline` section in `common-pipeline-conventions/SKILL.md` is the canonical home — it names the anti-pattern, lists the 6 parity-implying verbs (`match`, `rebuild`, `mirror`, `parity`, `make like`, `replicate`) each implying visual + structural + behavioral parity, classifies scope-narrowing as a DOMAIN gate, and instructs the agent to surface the scope decision via `AskUserQuestion` BEFORE starting work. The `prompt-refiner` agent gains a 6th grading axis `scope-fidelity` (weight 0.17; weights redistributed across all axes to sum to 1.0). The `bug-classifier` agent gains an action-verb interpretation section requiring `unclear` verdict on parity-verb prompts with narrower-than-literal interpretation. The `system-architect` Master Review Audit mode gains a scope-narrowing check in step 4 (the verdict JSON adds a `scope_fidelity_finding` block; populated `finding` is a verdict-failure condition). The Phase 2 architect brief mode gains a `Scope check` field in its Output structure. 35 new tests in `tests/test_scope_discipline.py`. **v1.3.0 makes auto-cleanup of merged worktrees the default for every `/architect-team` family invocation** — `/architect-team`, `/architect-team:bug-fix`, and `/architect-team:mini` each fire `cleanup_merged_worktrees()` as their FIRST action (before argument parsing, before refinement, before the v1.2.0 auto-worktree creation). Trigger 1 sweeps merged-and-forgotten worktrees from prior runs at the start of each new run; trigger 2 fires at the end of mini Phase M7 after the auto-merge to main and removes the mini's own worktree. The `exclude_current=True` safeguard means the cwd is never auto-removed (even on re-entry from a merged-branch worktree). The new helpers: `list_merged_architect_team_worktrees(against="origin/main", exclude_current=True) -> list[Path]` and `cleanup_merged_worktrees(against="origin/main", dry_run=False) -> list[Path]`. A new `/architect-team:cleanup-worktrees` command (the 12th) exposes the same helper for on-demand invocation with `--dry-run` and `--against <ref>` flags. Merged-branch detection uses `git merge-base --is-ancestor`; squash-merged branches are NOT auto-detected (false-negative is safer than false-positive). Best-effort: cleanup failures NEVER block the new run. **v1.2.0 makes auto-worktree creation the default** — `/architect-team`, `/architect-team:bug-fix`, and `/architect-team:mini` auto-create a fresh worktree at `<parent-of-repo>/<repo-name>-<slug>/` on branch `architect-team/<slug>` before invoking the pipeline skill, so the user's main checkout stays put and each run is filesystem-isolated. The v1.2.0 helper functions: `create_run_worktree(slug, base_branch="main", parent_dir=None)`, `cleanup_run_worktree(worktree_path, remove_branch=False)`, `current_worktree_is_run()` (re-entry detection — no nested worktrees), `current_run_slug()`. Collision handling appends `-2`, `-3`, ... when either path or branch is taken. The `--no-worktree` flag (or any natural-language opt-out — *"no worktree"*, *"don't create a worktree"*, *"single tree"*, *"in place"*) reverts to v1.1.0 single-tree behavior verbatim. **v1.1.0 closes the cross-session coordination gap for worktree-based parallel runs** via a 3-layer model: filesystem isolation via git worktrees (each session edits/tests/commits in its own working tree on its own branch), architectural coordination via the v1.0.0 `.architect-team/locks/` JSON lock layer resolved to the MAIN worktree, and context sharing via the MemPalace store also resolved to the MAIN worktree. **v1.0.0 made Claude Code's experimental Agent Teams primitive the default dispatch mode** — long-lived named teammates with their own 1M context windows, a shared task list, direct `SendMessage`, and a Lead that owns coordination. Teams mode requires the `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` env var (or the equivalent in `~/.claude/settings.json`) AND Claude Code ≥ 2.1.32 — without those, the pipeline silently falls back to subagents mode (the v0.10.0 behavior, unchanged). A `--no-teams` flag on any of the three pipeline commands forces subagents mode even when teams are available. Every nested-team pattern in the pipeline flattens to Lead-owned dispatch per the Agent Teams docs' "no nested teams" constraint. The plugin orchestrates **MemPalace wake-up → Phase −2 (Triage) → Phase −1 → 8** against a requirements folder, with sibling pipelines for **bug-fix** (`/architect-team:bug-fix`), **UX testing** (`/architect-team:ux-test`, v0.9.29), and **mini** (`/architect-team:mini`, v0.10.0).

## Conventions at a glance

- Author commits with explicit override (repo's local git config has a "Paul Ingrram" typo):
  `git -c user.name="Paul Ingram" -c user.email="paulingram@users.noreply.github.com" commit ...`
- Run the test suite via `python -m pytest -v` from repo root (2318 PASS + 1 SKIPPED expected as of v2.1.0).
- Bump version: update `.claude-plugin/plugin.json` + `.claude-plugin/marketplace.json` + `CHANGELOG.md`.
- Runtime state lives under `.architect-team/` (gitignored). v1.1.0 splits it into **shared** (in main worktree: `locks/`, `.mempalace/`, `run-history/`) and **per-run** (in current worktree: `reviews/`, `teammates/`, `handoffs/`, this-run's `openspec/changes/<slug>/`). The `scripts/setup/worktree_paths.py` helper resolves both via `shared_state_dir()` / `run_state_dir()`. **v1.2.0** auto-creates the run worktree itself (default; `--no-worktree` opts out) via `scripts/setup/worktree_lifecycle.py::create_run_worktree(slug)` — branch `architect-team/<slug>`, path `<parent-of-repo>/<repo-name>-<slug>/`, collision-handled by suffix-bumping `-2`, `-3`, ... **v1.3.0** auto-cleans merged worktrees on the next run via `cleanup_merged_worktrees()` fired as the first action of every `/architect-team` family invocation (and at the end of mini Phase M7 after the auto-merge), with `exclude_current=True` keeping the cwd safe; the new `/architect-team:cleanup-worktrees` command is the on-demand manual variant with `--dry-run` and `--against <ref>` flags.
