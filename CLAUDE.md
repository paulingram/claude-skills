# architect-team plugin

## Codebase Overview

This repo IS the source of the `architect-team` Claude Code plugin — a spec-to-production multi-agent coding pipeline. As of **v1.2.0** it ships **26 skills, 27 named agents, 11 slash commands**, 3 enforcement hooks (wired for both `PostToolUse(TaskUpdate)` / `SubagentStop` and the teams-mode `TaskCompleted` / `TeammateIdle` triggers), cross-platform setup scripts, an opt-in per-project email notifier, the v1.0.0 cross-session lock layer (`hooks/locks.py`), the v1.0.0 teams-mode detection helper (`scripts/setup/teams_mode.py`), the v1.1.0 worktree-aware state-resolution helper (`scripts/setup/worktree_paths.py`), and the new v1.2.0 worktree-lifecycle helper (`scripts/setup/worktree_lifecycle.py`). **v1.2.0 makes auto-worktree creation the default for every `/architect-team` family invocation** — `/architect-team`, `/architect-team:bug-fix`, and `/architect-team:mini` now auto-create a fresh worktree at `<parent-of-repo>/<repo-name>-<slug>/` on branch `architect-team/<slug>` before invoking the pipeline skill, so the user's main checkout stays put and each run is filesystem-isolated. The new helper exposes 4 functions: `create_run_worktree(slug, base_branch="main", parent_dir=None)`, `cleanup_run_worktree(worktree_path, remove_branch=False)`, `current_worktree_is_run()` (re-entry detection — no nested worktrees), `current_run_slug()`. Collision handling appends `-2`, `-3`, ... when either path or branch is taken. Cleanup is NOT automatic — the pipeline recommends `git worktree remove <path>` at Phase 8 / B8 / M7 success and the user decides when to clean up. The `--no-worktree` flag (or any natural-language opt-out — *"no worktree"*, *"don't create a worktree"*, *"single tree"*, *"in place"*) reverts to v1.1.0 single-tree behavior verbatim. **v1.1.0 closes the cross-session coordination gap for worktree-based parallel runs** via a 3-layer model: filesystem isolation via git worktrees (each session edits/tests/commits in its own working tree on its own branch), architectural coordination via the v1.0.0 `.architect-team/locks/` JSON lock layer resolved to the MAIN worktree, and context sharing via the MemPalace store also resolved to the MAIN worktree. **v1.0.0 made Claude Code's experimental Agent Teams primitive the default dispatch mode** — long-lived named teammates with their own 1M context windows, a shared task list, direct `SendMessage`, and a Lead that owns coordination. Teams mode requires the `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` env var (or the equivalent in `~/.claude/settings.json`) AND Claude Code ≥ 2.1.32 — without those, the pipeline silently falls back to subagents mode (the v0.10.0 behavior, unchanged). A `--no-teams` flag on any of the three pipeline commands forces subagents mode even when teams are available. Every nested-team pattern in the pipeline flattens to Lead-owned dispatch per the Agent Teams docs' "no nested teams" constraint. The plugin orchestrates **MemPalace wake-up → Phase −2 (Triage) → Phase −1 → 8** against a requirements folder, with sibling pipelines for **bug-fix** (`/architect-team:bug-fix`), **UX testing** (`/architect-team:ux-test`, v0.9.29), and **mini** (`/architect-team:mini`, v0.10.0).

**Stack:** Markdown (skills/agents/commands), JSON (plugin/marketplace/hooks metadata), Python 3.10+ (hooks + setup scripts + the v1.0.0 `scripts/setup/teams_mode.py` detector + the v1.0.0 `hooks/locks.py` cross-session lock layer + the v1.1.0 `scripts/setup/worktree_paths.py` worktree-aware state-resolution helper + the new v1.2.0 `scripts/setup/worktree_lifecycle.py` worktree-lifecycle helper + the stdlib-only `scripts/notify/` email notifier + 1702 pytest self-tests + 1 skipped).

**Structure:** `.claude-plugin/` (identity), `skills/` (26 dirs), `agents/` (27 files), `commands/` (11 files), `hooks/` (JSON wiring + 3 enforcement scripts + a shared `review_evidence_schema.py` module — review-gate evidence schema v6 — + the `locks.py` cross-session lock layer), `scripts/setup/` (`setup.py` + `install_mempalace.py` + `teams_mode.py` detector + the v1.1.0 `worktree_paths.py` resolution helper + the new v1.2.0 `worktree_lifecycle.py` lifecycle helper), `scripts/notify/` (`notify.py` — the best-effort, stdlib-only per-project email notifier), `tests/` (1702 pytest tests + 1 skipped across 74 test files with structural + cross-consistency coverage), `docs/` (CODEBASE_MAP.md, INTEGRATION_MAP.md, `superpowers/` historical design + plan).

For full architecture, file purposes, conventions, gotchas, and a navigation guide, see [`docs/CODEBASE_MAP.md`](docs/CODEBASE_MAP.md).

## Conventions at a glance

- Author commits with explicit override (repo's local git config has a "Paul Ingrram" typo):
  `git -c user.name="Paul Ingram" -c user.email="paulingram@users.noreply.github.com" commit ...`
- Run the test suite via `python -m pytest -v` from repo root (1702 PASS + 1 SKIPPED expected as of v1.2.0).
- Bump version: update `.claude-plugin/plugin.json` + `.claude-plugin/marketplace.json` + `CHANGELOG.md`.
- Runtime state lives under `.architect-team/` (gitignored). v1.1.0 splits it into **shared** (in main worktree: `locks/`, `.mempalace/`, `run-history/`) and **per-run** (in current worktree: `reviews/`, `teammates/`, `handoffs/`, this-run's `openspec/changes/<slug>/`). The `scripts/setup/worktree_paths.py` helper resolves both via `shared_state_dir()` / `run_state_dir()`. **v1.2.0** auto-creates the run worktree itself (default; `--no-worktree` opts out) via `scripts/setup/worktree_lifecycle.py::create_run_worktree(slug)` — branch `architect-team/<slug>`, path `<parent-of-repo>/<repo-name>-<slug>/`, collision-handled by suffix-bumping `-2`, `-3`, ...
