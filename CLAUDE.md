# architect-team plugin

## Codebase Overview

This repo IS the source of the `architect-team` Claude Code plugin — a spec-to-production multi-agent coding pipeline. As of **v1.6.0** it ships **26 skills, 27 named agents, 13 slash commands**, 3 enforcement hooks (wired for both `PostToolUse(TaskUpdate)` / `SubagentStop` and the teams-mode `TaskCompleted` / `TeammateIdle` triggers), cross-platform setup scripts, an opt-in per-project email notifier, the v1.0.0 cross-session lock layer (`hooks/locks.py`), the v1.0.0 teams-mode detection helper (`scripts/setup/teams_mode.py` — extended in v1.5.0 with `format_dispatch_banner()`), the v1.1.0 worktree-aware state-resolution helper (`scripts/setup/worktree_paths.py`), and the v1.2.0+v1.3.0 worktree-lifecycle helper (`scripts/setup/worktree_lifecycle.py` — 6 public functions now). **v1.6.0 adds the Teammate Git Discipline** — a real-world failure surfaced in a separate user session exposed a discipline gap: four teammates dispatched in parallel against the same working tree each ran `git stash` to verify their work against baseline; the concurrent stash + pop operations interleaved catastrophically, the reflog showed 10+ consecutive `reset: moving to HEAD` entries, and three of four teammates' work was lost (only the last writer survived). The plugin had no rule forbidding teammates from running destructive git operations, so the teammates did. v1.6.0 ships the discipline at four enforcement points (same shape as v1.4.0 scope-discipline): (1) the new `## Teammate git discipline` section in `common-pipeline-conventions/SKILL.md` is the canonical home — names the 6 forbidden destructive operations (`git stash` / `git stash pop`, `git reset --hard`, `git rebase`, `git commit --amend`, `git checkout <other-branch>` / `git checkout .`, `git clean -f`), documents the heirship-app-v2 worked example with the smoking-gun reflog signature (`reset: moving to HEAD` repeated 10×), and names the right pattern. (2) The 3 pipeline bodies each carry a v1.6.0 anti-pattern entry in `## Operating rules (non-negotiable)`. (3) All 27 `agents/*.md` files gain a `## Forbidden git operations` section as a uniform 5-line block between `## Operating context` and the next H2. (4) The `team-spawning-and-review-gates/SKILL.md` documents the `## Baseline SHA capture` orchestrator-side mechanics — the orchestrator runs `BASELINE_SHA=$(git rev-parse HEAD)` at run start, persists it to `intake-state.json` as `baseline_sha`, and carries it in every teammate's spawn brief (extending the v0.9.13 manifest schema); teammates use `git diff $BASELINE_SHA -- <my-files>` for baseline verification instead of `git stash`. 265 new tests in `tests/test_teammate_git_discipline.py` audit the structural discipline. **v1.5.0 added Dispatch-Mode Observability** — the user's direct question *"how do I know if a team is deployed via agent teams vs subagents, can we show an indicator"* exposed a real observability gap (v1.0.0 made the decision silent: it landed in `intake-state.json` but no user-visible signal surfaced). v1.5.0 ships three pieces: (1) the new `format_dispatch_banner()` helper renders a one-block banner naming the active mode + (in the fallback case) the diagnosed reason; (2) all 3 pipeline-driving slash commands now print the banner as their FIRST user-visible action (before the v1.3.0 auto-cleanup, before argument parsing); (3) the new `/architect-team:status` command (13th) reports dispatch mode + active worktrees + open SRs + last completed run; (4) the Phase 8 / B8 / M7 commit-message templates gain a `Dispatch-Mode: <teams|subagents>` trailer so `git log --format=%(trailers)` can answer "which mode produced this commit?" archeologically. The banner is informational, never gating — subprocess failure surfaces a one-line note and the run continues. The dispatch decision itself is unchanged from v1.0.0. **v1.4.0 added the Scope Discipline** — a user-reported plugin-level gap: agents using the package were silently NARROWING the user's prompt at intake (e.g., *"match the oracle"* read as *"enrichment + hardcoded data purge"* with the visual rebuild deferred). The v0.9.36 anti-deferral discipline forbade the MID-RUN version; v1.4.0 extends the forbiddance to INTAKE. The new `## Scope discipline` section in `common-pipeline-conventions/SKILL.md` is the canonical home — it names the anti-pattern, lists the 6 parity-implying verbs (`match`, `rebuild`, `mirror`, `parity`, `make like`, `replicate`) each implying visual + structural + behavioral parity, classifies scope-narrowing as a DOMAIN gate, and instructs the agent to surface the scope decision via `AskUserQuestion` BEFORE starting work. The `prompt-refiner` agent gains a 6th grading axis `scope-fidelity` (weight 0.17; weights redistributed across all axes to sum to 1.0). The `bug-classifier` agent gains an action-verb interpretation section requiring `unclear` verdict on parity-verb prompts with narrower-than-literal interpretation. The `system-architect` Master Review Audit mode gains a scope-narrowing check in step 4 (the verdict JSON adds a `scope_fidelity_finding` block; populated `finding` is a verdict-failure condition). The Phase 2 architect brief mode gains a `Scope check` field in its Output structure. 35 new tests in `tests/test_scope_discipline.py`. **v1.3.0 makes auto-cleanup of merged worktrees the default for every `/architect-team` family invocation** — `/architect-team`, `/architect-team:bug-fix`, and `/architect-team:mini` each fire `cleanup_merged_worktrees()` as their FIRST action (before argument parsing, before refinement, before the v1.2.0 auto-worktree creation). Trigger 1 sweeps merged-and-forgotten worktrees from prior runs at the start of each new run; trigger 2 fires at the end of mini Phase M7 after the auto-merge to main and removes the mini's own worktree. The `exclude_current=True` safeguard means the cwd is never auto-removed (even on re-entry from a merged-branch worktree). The new helpers: `list_merged_architect_team_worktrees(against="origin/main", exclude_current=True) -> list[Path]` and `cleanup_merged_worktrees(against="origin/main", dry_run=False) -> list[Path]`. A new `/architect-team:cleanup-worktrees` command (the 12th) exposes the same helper for on-demand invocation with `--dry-run` and `--against <ref>` flags. Merged-branch detection uses `git merge-base --is-ancestor`; squash-merged branches are NOT auto-detected (false-negative is safer than false-positive). Best-effort: cleanup failures NEVER block the new run. **v1.2.0 makes auto-worktree creation the default** — `/architect-team`, `/architect-team:bug-fix`, and `/architect-team:mini` auto-create a fresh worktree at `<parent-of-repo>/<repo-name>-<slug>/` on branch `architect-team/<slug>` before invoking the pipeline skill, so the user's main checkout stays put and each run is filesystem-isolated. The v1.2.0 helper functions: `create_run_worktree(slug, base_branch="main", parent_dir=None)`, `cleanup_run_worktree(worktree_path, remove_branch=False)`, `current_worktree_is_run()` (re-entry detection — no nested worktrees), `current_run_slug()`. Collision handling appends `-2`, `-3`, ... when either path or branch is taken. The `--no-worktree` flag (or any natural-language opt-out — *"no worktree"*, *"don't create a worktree"*, *"single tree"*, *"in place"*) reverts to v1.1.0 single-tree behavior verbatim. **v1.1.0 closes the cross-session coordination gap for worktree-based parallel runs** via a 3-layer model: filesystem isolation via git worktrees (each session edits/tests/commits in its own working tree on its own branch), architectural coordination via the v1.0.0 `.architect-team/locks/` JSON lock layer resolved to the MAIN worktree, and context sharing via the MemPalace store also resolved to the MAIN worktree. **v1.0.0 made Claude Code's experimental Agent Teams primitive the default dispatch mode** — long-lived named teammates with their own 1M context windows, a shared task list, direct `SendMessage`, and a Lead that owns coordination. Teams mode requires the `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` env var (or the equivalent in `~/.claude/settings.json`) AND Claude Code ≥ 2.1.32 — without those, the pipeline silently falls back to subagents mode (the v0.10.0 behavior, unchanged). A `--no-teams` flag on any of the three pipeline commands forces subagents mode even when teams are available. Every nested-team pattern in the pipeline flattens to Lead-owned dispatch per the Agent Teams docs' "no nested teams" constraint. The plugin orchestrates **MemPalace wake-up → Phase −2 (Triage) → Phase −1 → 8** against a requirements folder, with sibling pipelines for **bug-fix** (`/architect-team:bug-fix`), **UX testing** (`/architect-team:ux-test`, v0.9.29), and **mini** (`/architect-team:mini`, v0.10.0).

**Stack:** Markdown (skills/agents/commands), JSON (plugin/marketplace/hooks metadata), Python 3.10+ (hooks + setup scripts + the v1.0.0 `scripts/setup/teams_mode.py` detector — extended in v1.5.0 with `format_dispatch_banner()` — + the v1.0.0 `hooks/locks.py` cross-session lock layer + the v1.1.0 `scripts/setup/worktree_paths.py` worktree-aware state-resolution helper + the v1.2.0+v1.3.0 `scripts/setup/worktree_lifecycle.py` worktree-lifecycle helper + the stdlib-only `scripts/notify/` email notifier + 2030 pytest self-tests + 1 skipped).

**Structure:** `.claude-plugin/` (identity), `skills/` (26 dirs), `agents/` (27 files), `commands/` (13 files — `status.md` joins in v1.5.0), `hooks/` (JSON wiring + 3 enforcement scripts + a shared `review_evidence_schema.py` module — review-gate evidence schema v6 — + the `locks.py` cross-session lock layer), `scripts/setup/` (`setup.py` + `install_mempalace.py` + `teams_mode.py` detector — extended in v1.5.0 with `format_dispatch_banner()` — + the v1.1.0 `worktree_paths.py` resolution helper + the v1.2.0+v1.3.0 `worktree_lifecycle.py` lifecycle helper), `scripts/notify/` (`notify.py` — the best-effort, stdlib-only per-project email notifier), `tests/` (2030 pytest tests + 1 skipped across 78 test files with structural + cross-consistency coverage), `docs/` (CODEBASE_MAP.md, INTEGRATION_MAP.md, `superpowers/` historical design + plan).

For full architecture, file purposes, conventions, gotchas, and a navigation guide, see [`docs/CODEBASE_MAP.md`](docs/CODEBASE_MAP.md).

## Conventions at a glance

- Author commits with explicit override (repo's local git config has a "Paul Ingrram" typo):
  `git -c user.name="Paul Ingram" -c user.email="paulingram@users.noreply.github.com" commit ...`
- Run the test suite via `python -m pytest -v` from repo root (2030 PASS + 1 SKIPPED expected as of v1.6.0).
- Bump version: update `.claude-plugin/plugin.json` + `.claude-plugin/marketplace.json` + `CHANGELOG.md`.
- Runtime state lives under `.architect-team/` (gitignored). v1.1.0 splits it into **shared** (in main worktree: `locks/`, `.mempalace/`, `run-history/`) and **per-run** (in current worktree: `reviews/`, `teammates/`, `handoffs/`, this-run's `openspec/changes/<slug>/`). The `scripts/setup/worktree_paths.py` helper resolves both via `shared_state_dir()` / `run_state_dir()`. **v1.2.0** auto-creates the run worktree itself (default; `--no-worktree` opts out) via `scripts/setup/worktree_lifecycle.py::create_run_worktree(slug)` — branch `architect-team/<slug>`, path `<parent-of-repo>/<repo-name>-<slug>/`, collision-handled by suffix-bumping `-2`, `-3`, ... **v1.3.0** auto-cleans merged worktrees on the next run via `cleanup_merged_worktrees()` fired as the first action of every `/architect-team` family invocation (and at the end of mini Phase M7 after the auto-merge), with `exclude_current=True` keeping the cwd safe; the new `/architect-team:cleanup-worktrees` command is the on-demand manual variant with `--dry-run` and `--against <ref>` flags.
