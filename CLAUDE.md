# CLAUDE TEAM SIX (CT6)

> Internal plugin slug: `architect-team` (preserved for backward compatibility with existing installations + the 20 slash commands). CLAUDE TEAM SIX is the user-facing display name introduced in v2.2.0.

## What this repo is

This repo IS the source of the **CLAUDE TEAM SIX** Claude Code plugin (internal slug `architect-team`) — a spec-to-production multi-agent coding pipeline. It orchestrates **MemPalace wake-up → Phase −2 (Triage) → Phase −1 → 8** against a requirements folder, with sibling pipelines for **bug-fix** (`/architect-team:bug-fix`), **UX testing** (`/architect-team:ux-test`), and **mini** (`/architect-team:mini`).

As of **v3.15.0** it ships **41 skills + the v2.3.0 phenotype subsystem (now 4 seeded phenotypes — `user-management`, `config-management`, `ai-management`, `code-wiki`), 37 named agents, 20 slash commands, and 4501 pytest self-tests + 5 skipped** (across 178 test files; green under both Windows cp1252 and `PYTHONUTF8=1`). The default dispatch mode is Claude Code's experimental **Agent Teams** primitive (long-lived named teammates with their own 1M context windows, a shared task list, direct `SendMessage`, and a Lead that owns coordination); it requires `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` and Claude Code ≥ 2.1.32, and silently falls back to subagents mode otherwise. `--no-teams` forces subagents mode.

## Stack

- `hooks/` — JSON wiring + 5 enforcement scripts (`review-gate-task.py` PostToolUse(TaskUpdate)+TaskCompleted / `teammate-idle-check.py` SubagentStop+TeammateIdle / `pipeline-completion-audit.py` Stop / `pretool_unilateral_override_guard.py` PreToolUse(Edit/Write/NotebookEdit) / `pretool_skill_gate.py` PreToolUse[*] — v3.15.0 real-time skill-invocation hard-gate) + the shared `review_evidence_schema.py` (evidence schema **v7**: 17 required fields incl. 5 VAO fields + 3 optional fields `interactions_honored_review` / `live_verification_review` / `appearance_scope_review`) + `locks.py` (cross-session lock layer) + the **`hooks/vao/` package** (the 20 deterministic Layer-3 verification tools, split per discipline family; see CODEBASE_MAP) behind the **`hooks/vao_tools.py` facade** (re-exports every test-referenced name + keeps the CLI byte-identical) + `shared_util.py` (`load_json` / `_utc_now_iso` / the unified JSONL reader) + `shared_rule_constants.py` + `override_markers.py` + `skill_invocation_audit.py` (Layer 6 Stop-hook auditor; its detection — `find_skill_requests` + `COMMAND_TO_SKILLS` — is reused by the v3.15.0 `pretool_skill_gate.py` real-time gate) + `inflight_inbox.py` + `discipline_registry.py` + `run_metrics.py` + `lineage_graph.py`.
- `scripts/notify/notify.py` — the best-effort, stdlib-only per-project email notifier (6 event types incl. `heartbeat`).
- `scripts/phenotypes/phenotypes.py` — the deterministic phenotype engine.

The 20 Layer-3 verification CLI subcommands (all dispatched through the `hooks/vao_tools.py` facade): `verify-oracle-match`, `verify-baseline-clean`, `verify-no-fake-data`, `verify-every-element`, `verify-rendered-parity`, `verify-interactions-honored`, `verify-live-verification-claim`, `verify-live-data-wiring`, `verify-no-standing-red`, `verify-no-end-of-run-deferral`, `verify-per-persona-path-coverage`, `verify-affordance-coverage`, `verify-no-implementation-scope-cut`, `verify-test-prod-safety-classification`, `verify-discipline-registry-current`, `verify-inflight-clarifications-processed`, `verify-deploy-mandate-satisfied`, `verify-target-element-measured`, `verify-no-pipeline-bypass`, `verify-no-unilateral-override`.

## Conventions at a glance

- Author commits with explicit override (repo's local git config has a "Paul Ingrram" typo):
  `git -c user.name="Paul Ingram" -c user.email="paulingram@users.noreply.github.com" commit ...`
- Run the test suite via `python -m pytest -v` from repo root (green under both Windows cp1252 and `PYTHONUTF8=1`; the expected pass/skip count for the current release is in the CHANGELOG entry).
- Bump version: update `.claude-plugin/plugin.json` + `.claude-plugin/marketplace.json` + `CHANGELOG.md`.
- Runtime state lives under `.architect-team/` (gitignored). v1.1.0 splits it into **shared** (in main worktree: `locks/`, `.mempalace/`, `run-history/`) and **per-run** (in current worktree: `reviews/`, `teammates/`, `handoffs/`, this-run's `openspec/changes/<slug>/`). The `scripts/setup/worktree_paths.py` helper resolves both via `shared_state_dir()` / `run_state_dir()`. **v1.2.0** auto-creates the run worktree (default; `--no-worktree` opts out) via `scripts/setup/worktree_lifecycle.py::create_run_worktree(slug)` — branch `architect-team/<slug>`, path `<parent-of-repo>/<repo-name>-<slug>/`. **v1.3.0** auto-cleans merged worktrees on the next run via `cleanup_merged_worktrees()` (with `exclude_current=True` keeping the cwd safe); the `/architect-team:cleanup-worktrees` command is the on-demand variant. **v3.7.0** auto-merges a clean run's branch `--no-ff` into `main` and prunes it (`--no-auto-merge` opts out; a conflict falls back to a PR, never `--force`).
