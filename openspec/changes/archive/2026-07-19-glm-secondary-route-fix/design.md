# Design — glm-secondary-route-fix

## Root cause

Two independent, stacked defects (both live-verified 2026-07-17; see `.architect-team/bug-isolation/glm-secondary-route-fix/`):

- **BUG-A (gateway route dialect).** `SECONDARY_PROVIDERS` encodes each provider's route as a full `route_model` string with a hard-coded `openai/` prefix (`scripts/setup/set_default_model.py:76,83`); `_secondary_route_model` / `build_gateway_config` (`scripts/setup/install_gateway.py:538,543`) pass it through. LiteLLM's *proxy* drives Anthropic-format (`/v1/messages`) traffic for `openai/*` models through the OpenAI **Responses API**; api.openai.com implements `/responses` (codex-gen models require it), api.z.ai implements only `/chat/completions` → every call 404s at `/v4/responses`. Discriminant executed: Z.ai direct 200; `hosted_vllm/` dialect through the same gateway 200 (thinking blocks intact); SDK and Router paths 200 on BOTH prefixes (the defect is proxy-bridge-specific, hence invisible to any non-proxy test).
- **BUG-B (harness spawn gate).** Claude Code's Agent-Teams spawn path (current build; changed after 2026-07-16) validates teammate model ids client-side — a spawn on `ct6-secondary` emits ZERO gateway requests (gateway-log verified). Any frontmatter alias that is not a known Claude model id never reaches the gateway, regardless of dialect.
- **Compounding (wrong-hop confirmation).** `confirm_gateway_serving` (`install_gateway.py:1040`) probes `/v1/models` only (lines 1026-1029) — model *listing*, not completion *serving* — so setup printed "CONFIRMED live — CT6 runs the split" while the completion hop 404'd.

## Proposed fix (class-scoped)

The class: **provider-agnostic route emission that assumes every OpenAI-compatible endpoint speaks the full OpenAI API, and a split alias that assumes the harness passes custom model ids through.** The fix removes both assumptions for EVERY current and future provider entry, not just zai:

1. **Registry route dialect (BUG-A).** Each `SECONDARY_PROVIDERS` entry gains a required `route_dialect` field naming the LiteLLM provider prefix the gateway route uses: `openai` → `openai` (Responses-API dialect), `zai` → `hosted_vllm` (strict chat-completions). `route_model` becomes derived (`<route_dialect>/<model>`) or is retained but validated consistent — implementation picks the cleaner shape; the extensibility pin REQUIRES the field so a future entry cannot omit it. `_secondary_route_model` + `build_gateway_config` consume the registry field; no hard-coded prefix remains in the generator.
2. **Spawn-compatible impersonation alias (BUG-B).** `install_gateway.py` exports `SPAWN_ALIAS_MODEL_ID = "claude-haiku-4-5"` (a real, harness-accepted Claude id). The generator emits an explicit route `model_name: claude-haiku-4-5 → <dialect>/<secondary model>` AHEAD of the anthropic catch-all (route order is already explicit-first). The split (`apply_split`) writes `SPAWN_ALIAS_MODEL_ID` into the 21 dev-class agents' frontmatter instead of `ct6-secondary`; the `ct6-secondary` route REMAINS served for direct/scripted callers and as the state-recorded alias lineage. `gateway.json` records `{"spawn_alias": "claude-haiku-4-5", "spawn_alias_maps_to": "glm-5.2"}`; `status` prints the disclosure line; `maybe_heal_model_split` heals dev-class frontmatter to the spawn alias (its heal target follows the gateway-recorded state, extended with the new field). **Collision safeguard (refiner gap 1):** `claude-haiku-4-5`'s organic-use check — the machine's gateway log shows zero organic requests for that id (fable is the only session model; verified in `gateway-4000.log`), and the disclosure line makes the rewrite discoverable; the id is test-pinned so changing it is a deliberate act. Trade-off recorded: requests labeled `claude-haiku-4-5` through this gateway are served by GLM 5.2 — disclosed in status + README, the standard LiteLLM model-impersonation pattern for harness-constrained callers.
3. **Honest confirmation (wrong-hop).** `confirm_gateway_serving` gains a `/v1/messages` completion probe (max_tokens ≤ 32, asserts a non-empty content block) against the state-recorded spawn alias (and falls back to `ct6-secondary`), behind the SAME injectable transport seam the /v1/models probe uses today (keyless CI stays green; the live probe runs only where a gateway + keys exist). The success message reports BOTH hops honestly: gateway-serving CONFIRMED (completion returned) + spawn-hop boundary note (verifiable only from a fresh harness session).
4. **Docs.** CHANGELOG new-entry FO-1 supersedure (two-bug RCA, never editing archived entries); README gateway section: impersonation disclosure + the prisma no-DB auth-500 troubleshooting note. Fix path B (`claude-ct6-secondary` claude-shaped alias) documented as a future toggle; path C (upstream custom-id support) as a follow-up note.
5. **Machine application (Phase B5 deploy).** Re-run the repo-source installer against `~/.architect-team/gateway` (regenerating config.yaml, replacing the hand-edit incl. the leftover `claude-ct6-secondary` test route), re-apply the split to the installed plugin copy, restart the gateway, and run the upgraded probe: a real GLM completion via `/v1/messages` is the deploy-green criterion.

## Iteration 2 (SR-glm-fix-iter2 — B6 witness findings)

The first deploy exposed two further members of the same wrong-hop class, both live-verified:

6. **Restart-on-regeneration (installer).** The register/start step is start-if-not-running: after `config.yaml` regeneration the RUNNING process kept serving the old config (stale-process marker `claude-ct6-secondary` observed via `/model/info`). A config-regenerating install SHALL restart a running gateway process (bounded stop→start with the existing poll-until-healthy), or equivalently the confirm step SHALL detect a served-config mismatch and trigger the existing one-restart repair cycle. Detection seam: the generated config's model-group set vs the served `/model/info` group set.
7. **Upstream-verifying probe.** The `/v1/messages` completion probe SHALL verify the serving deployment, not just receive a completion: match the response's `x-litellm-model-id` against `/model/info` and assert the deployment's `litellm_params.model` equals the registry-derived dialect route (`hosted_vllm/glm-5.2` for zai). A completion served by any other deployment (e.g. the `anthropic/*` wildcard's dynamically-instantiated real-Haiku deployment) FAILS the probe with the observed upstream named. This closes the probe's last dishonesty window: silently billing Anthropic while reporting the split live.

## Iteration 3 (SR-glm-fix-iter3 — the fallback-green + file-vs-served staleness escapes)

8. **Spawn-alias-mandatory confirm.** When `gateway.json` records a `spawn_alias`, the confirm SHALL verify completion + upstream identity ON that alias; failure or mismatch there FAILS the confirm (repair cycle fires). The ct6-secondary fallback green is permitted ONLY when no spawn alias is recorded (legacy states).
9. **Served-state staleness detection.** Staleness SHALL be judged against the SERVED state, not the prior file: compare the generated config's model-group set + the spawn route's expected deployment against `/model/info`; mismatch ⇒ restart. This catches file-current-process-stale — the state the byte-diff cannot see.

10. **Iteration 4 (SR-glm-fix-iter4) — restart stops the port-holder and verifies the bind.** The stop half SHALL resolve the LISTENING pid on the gateway port and stop that process when tracked stopping leaves the port held (localhost gateway port, owner-consented restart; honest report if refused). The start half SHALL verify the new instance actually binds (served state matches the generated config) before reporting restarted; a held port never leaks doomed instances.

11. **Iteration 5 (SR-glm-fix-iter5, user mandate) — exactly-one-instance + owned start + enforced auth.** Install SHALL stop every gateway instance belonging to this state dir (config-path cmdline match) before its one start; register SHALL NOT blind-start over a serving instance; the post-start port-holder SHALL be verified as the just-launched instance; confirm SHALL assert auth is enforced — an invalid key must be DENIED SERVICE: a 4xx is clean enforcement; a crash-5xx is enforcement-with-a-named-warning (the LiteLLM no-DB prisma quirk, README-cross-referenced) and does NOT fail confirm; only a 200 (service granted to a bad key) fails confirm and the master key SHALL never rotate on re-install (pinned).

12. **Iteration 6 (SR-glm-fix-iter6, user mandate 2 — the offline incident) — verify-then-swap, never dark.** The restart/deploy path SHALL prove the replacement before touching the serving instance: launch a STAGING instance of the identical generated config on a staging port, run the full verification ladder against it (bind, served-state, real-key completion + upstream identity, auth enforcement — which structurally proves env resolution incl. the master key), and ONLY on staging-green stop the live instance and start its replacement (minimized dark window, bounded post-cutover ownership + ladder re-verification with retry). Any staging failure leaves the old instance serving and fails the install honestly. `run_gateway.bat` SHALL refuse to launch when the gateway port is already LISTENING and SHALL log masked master-key presence before exec (incident-report recommendation). The incident's recovery procedure is codified in README troubleshooting + confirm failure hints. The STAGING instance SHALL be stopped on ALL exit paths (green cutover, ladder failure, exception — try/finally semantics, test-pinned), never reintroducing instance accumulation on the staging port; staging SHALL launch via the same env-sourcing path as the cutover start so staging-green predicts the replacement's env; the bat port guard SHALL key on the port THIS launch binds (so legitimate staging launches are never refused); retry-exhaustion failures SHALL embed the recovery procedure. Rationale: every active Claude session routes through this port — a deploy that can go dark or lose a port race takes the user offline (the 2026-07-18 incident).

13. **Iteration 7 (SR-glm-fix-iter7, CRITICAL) — tree-aware instances, route-aware staleness.** (a) An owned INSTANCE is a process TREE: enumeration groups owned pids by parent chain; stop semantics are TREE semantics with two distinct predicates (pin 1): the SWEEP is port-holder-tree IMMUNE (never touches any pid in the serving tree), while the deliberate cutover/repair stop acts on the holder's WHOLE tree (root-first/job-kill, no respawn-capable orphan) — both directions test-pinned (the cmd→litellm.exe→python.exe chain is the pinned replay). (b) Served-state staleness compares each served group's SERVING ROUTE against the generated config's route for that group name (catch-all expansion included) — wildcard-advertised anthropic/<id> groups on a fresh instance are routed, not stale; a group whose serving route contradicts the config (the true marker) still fails; and the check is TWO-ARMED (pin 2): config→served completeness also holds — a served state missing any explicitly-routed group of the generated config (e.g. a newly added spawn alias absorbed by the catch-all on a stale process) is stale. Wildcard patterns are derived from the generated config at comparison time; explicit routes take precedence over catch-all expansion. The deploy-6 forensics are replayed as tests; the live deploy re-runs only after the user-visible moratorium note in the final deploy step.
