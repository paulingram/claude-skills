# design — secondary-provider-registry

## Pinned design decisions (from the refined brief's open questions)

- **Neutral alias**: `ct6-secondary` (reads as a model id, greppable, provider-free). Constant `SECONDARY_ALIAS` in the lever; `CODEX_MODEL = "codex-5.6-sol"` RETAINED as `LEGACY_SECONDARY_ALIASES = ("codex-5.6-sol",)` for migration detection only.
- **Registry home**: `scripts/setup/set_default_model.py` (the lever) — `SECONDARY_PROVIDERS: dict[str, dict]` with per-entry keys `model` (upstream id), `key_env` (e.g. `ZAI_API_KEY`), `route_model` (LiteLLM `model:` value, e.g. `openai/gpt-5.6-sol` or `openai/glm-5.2`), `api_base` (None for native providers; `https://api.z.ai/api/paas/v4` for zai), `label` (human text for the popup). `install_gateway.py` imports it via the existing `_lever` load — the same cross-module single-sourcing as `CODEX_ALIAS` today.
- **Flags/env**: installer `--secondary <name>` / `--secondary-model <id>` / `--zai-key <key>` / `--re-ask-provider`; `--openai-model` deprecated synonym (maps onto the openai entry's model with a one-line stderr note). setup.py passes `--secondary` through; env `CT6_SECONDARY_PROVIDER` (absent = no signal — the CT6_* tri-state convention; an unknown value degrades to a warn row, never gates).
- **State**: gateway.json gains `secondary_provider` + `secondary_model` + `secondary_alias`; `codex_alias` key retired (written as `secondary_alias`; readers accept both). **Grandfathering rule**: state with `openai_model` and NO `secondary_provider` ⇒ inferred `openai` (recorded on the next state write); the popup/prompt NEVER fires for an inferred-or-recorded choice absent `--re-ask-provider`.
- **Choice resolution ladder** (mirrors key resolution): `--secondary` flag > `CT6_SECONDARY_PROVIDER` env > recorded gateway.json choice > grandfather inference > interactive ask (TTY prompt or wrapper question) > default `openai` (non-interactive first install with no signal — recorded, so it never re-asks). An interrupt (Ctrl-C/EOF) at the interactive provider prompt deliberately selects the ladder's FINAL rung — default `openai`, recorded like any default — the same interrupt-means-skip semantic as the key prompt (pinned by test; ADV3-7b).
- **Key slots become registry-driven**: slots = `anthropic` (always) + the CHOSEN provider's slot (`openai`→`OPENAI_API_KEY`, `zai`→`ZAI_API_KEY`). key-declines.json keys by slot name (existing shape holds; `zai` is a new slot name). Only the chosen provider's key gates enablement. **Key retention on rewrite (ADV3-2)**: every gateway.env rewrite MERGES — it retains every registry provider's existing stored key (plus `ANTHROPIC_API_KEY` and the master key, as the baseline did), new resolutions winning — so switching `--secondary` never deletes the other provider's stored key and switching back re-enables with no re-supply.
- **Config generation**: one route `model_name: ct6-secondary` → the chosen entry's `route_model` (+ `api_base` when set) + `api_key: os.environ/<key_env>`; the Anthropic explicit routes + catch-all are UNCHANGED (api-key mode) / absent (subscription) — provider choice affects only the secondary route.
- **Confirmation**: a fresh `install` expects `[ct6-secondary]` (+ `claude-fable-5` api-key) — provider-independent by construction since the just-regenerated config serves the alias. `status --live` expects the STATE-RECORDED alias (`secondary_alias`, falling back to legacy `codex_alias`) so a working legacy install probes green (ADV3-3). `status`/`install` summaries add `secondary=<provider>/<model>`.
- **Split/lever**: `apply_split` writes `ct6-secondary`; `policy_state` returns `codex-split`… renamed to `secondary-split` with `codex-split` accepted as a legacy synonym in readers (state files, uninstall guard, self-heal) so mixed-version states never wedge. `--split codex` kept as a deprecated synonym of `--split secondary`. `AGENT_ROLES` untouched.
- **Migration**: (a) EVERY `install` run that regenerates the gateway config — `setup`, `install --activate`, or a plain non-activate `install` (amended per ADV3B-1) — migrates an on-disk legacy split to the neutral alias in the same operation (idempotent; `--activate` additionally re-applies the full split); install time is the ONLY migration point to the neutral alias; (b) **heal-to-recorded-alias** (amended per adversarial finding ADV3-1): `maybe_heal_model_split` restores the split to the alias the gateway STATE records as served (`secondary_alias` > legacy `codex_alias` > no-op; whitespace-trimmed per ADV3B-2) and never writes an alias the recorded config doesn't route — an agent-side-only migration to `ct6-secondary` would desync the agents from a still-running v3.39 gateway that routes only `codex-5.6-sol` (every dev/checking/testing request would 404); the legacy-retention heal note names the `install --activate` migration path. gateway.json policy readers accept `codex-split` + `secondary-split`; writers write `secondary-split`.
- **Uninstall**: restores uniform fable from EITHER split alias; removes the provider choice only under `--purge`.

## Adversarial round 3 decisions (ADV3)

- **heal-to-recorded-alias** — adversarial finding ADV3-1: the SessionStart heal restores the split to the alias the gateway STATE records as served; an agent-side-only migration to the neutral alias would desync the serving side (a v3.39 config routes only `codex-5.6-sol`, so migrated agents would 404). Migration to the neutral alias happens at install time, which regenerates the config.
- **gateway.env merge-on-rewrite** — ADV3-2: retain all registry providers' stored keys + Anthropic + master on every rewrite; new resolutions win.
- **status --live expects the recorded alias** — ADV3-3: same principle as ADV3-1 applied to the probe expectation; a working legacy install probes green.
- **Provider-neutral setup help** — ADV3-4: `--external-llm`/`--codex` help names the `ct6-secondary` slot + the chosen provider, never the legacy `codex-5.6-sol → OpenAI` phrasing; provider-name enumerations in help derive from `sorted(SECONDARY_PROVIDERS)` (setup.py falls back to the shipped names if the lever load fails — help must never break).
- **Policy strings single-sourced** — ADV3-5: the lever exports `POLICY_SECONDARY_SPLIT` / `POLICY_UNIFORM_FABLE` / `LEGACY_POLICY_CODEX_SPLIT`; the gateway imports them; the SessionStart hook keeps a guarded local copy (it loads the lever dynamically only after its cheap state check; fail-open).

## Adversarial round 3b decisions (ADV3B)

- **Non-activate reinstall consistency + state carry-forward** — ADV3B-1 (MAJOR): the baseline state downgrade (a plain re-install recording `activated=false` + `uniform-fable`) was harmless while the regenerated config kept the SAME alias; this change's alias flip weaponized it — a plain `install` over a legacy-activated machine regenerated the config to route only `ct6-secondary` while the agents kept `codex-5.6-sol` (every dev/checking/testing request 404s), the state downgrade DISARMED the SessionStart self-heal, and `status --live` false-confirmed. Fix (the adversary's options b+c combined, orchestrator-sanctioned): (1) **state carry-forward** — a non-activate install carries the prior state's `activated` and `model_policy` forward via a merge-preserving write (this run neither activated nor deactivated anything); uninstall — and a future explicit deactivate — is the ONLY downgrade path, which keeps the heal armed and status truthful and also closes ADV3B-5 (the modern-machine heal-disarm on a plain re-install); (2) **serving-side consistency at every install exit** — whenever install regenerates config.yaml, any on-disk legacy split in the runtime agents dir is migrated to the neutral alias in the SAME run, regardless of `--activate` (migration of an already-activated machine's split is consistency maintenance, not a new activation; `migrate_legacy_split` rewrites only legacy-alias files, so agents with no split on disk are never touched and a fresh plain install stays split-neutral); `secondary_alias` records what the regenerated config routes. Net invariant (test-pinned): after ANY install over ANY prior state, the agents' split alias (when a split exists) is routed by the just-written config, the recorded state matches reality, `status --live` is truthful, and the self-heal stays armed on activated machines.
- **Whitespace-only recorded aliases read as absent** — ADV3B-2 (NOTE): both recorded-alias readers (the heal and the `status --live` expectation, the latter via the shared `_recorded_secondary_alias` helper) trim each candidate BEFORE the truthiness check, so a corrupt whitespace `secondary_alias` never masks a valid legacy `codex_alias` nor becomes a literal-whitespace probe expectation; an all-whitespace record reads as absent (heal fail-open no-op).

## Reuse Decisions

- No new modules/files in `scripts/` or `hooks/` — every change extends existing functions in place (`build_gateway_config`, `resolve_keys`, `_cmd_install/_cmd_status/_cmd_uninstall`, `setup_entry`, `apply_split`/`policy_state`, `maybe_heal_model_split`). Registry + alias constants extend `set_default_model.py` (extend > build-new; CODEBASE_MAP Stack lines confirm both files are the sanctioned homes).
- No new third-party dependency: the zai route uses LiteLLM's existing openai-compatible `api_base` form — no new provider package (comparison: a `zhipuai` SDK dependency was rejected; the gateway already speaks OpenAI-compatible HTTP).
- 21 `agents/*.md`: frontmatter-only rewrites via the existing lever machinery (no new writer).
- Tests extend the four existing test files; no new test file (the registry cases slot into `test_set_default_model.py` / `test_install_gateway.py`).
- `commands/architect-team-setup.md`: extends the existing External-LLM + ask-for-keys sections.

## Dev Environment

N/A — plugin repo; verification = hermetic pytest (both encodings) + the live gateway on THIS machine for the openai path. HONEST BOUNDARY: the zai path's LIVE end-to-end confirmation requires a Z.ai API key; without one at run time, zai coverage is hermetic (config-shape + registry + prompt/decline + migration tests with stubbed probers), and the live confirm runs when a user supplies the key — exactly the provisioned-but-not-enabled posture the gateway already documents.

## Error handling

- Unknown `--secondary`/env value → warn row naming valid entries; never gates; no state written.
- Missing chosen-provider key → the honest provisioned-but-NOT-enabled state + v3.38.0 ask-then-apply prompting (the zai slot behaves exactly like openai's today).
- Legacy-alias states never wedge: every reader accepts old + new alias/policy strings; only writers are new-only.

## Testing

- Lever: registry shape pins; apply_split writes ct6-secondary; policy_state recognizes new + legacy splits; migration rewrite; deprecated `--split codex` synonym.
- Gateway: provider resolution ladder (flag/env/state/grandfather/default) incl. never-re-asks-when-recorded + `--re-ask-provider`; zai config shape (route/api_base/env-ref; raw key nowhere but gateway.env); zai key slot prompt/decline; grandfather inference; confirm expects the alias; uninstall restores from either alias; setup pass-through; deprecated `--openai-model` synonym.
- SessionStart: heal restores the STATE-RECORDED alias in both directions (legacy state retains `codex-5.6-sol` + names the `install --activate` migration; `secondary_alias` state heals to `ct6-secondary`); agents already matching the recorded alias are a silent no-op; the healed agents-side alias always equals the state-recorded served alias (serving-side consistency pin); no recorded alias => no-op; state-policy legacy string accepted.
- test_agents.py `VALID_MODELS`: gains `ct6-secondary`, keeps `codex-5.6-sol` (legacy-valid), ship state pin stays uniform fable.
- Hermeticity: all new tests ride the existing autouse scrubs (registry env, live-probe stub, runner stub) + scrub `CT6_SECONDARY_PROVIDER` and `ZAI_API_KEY`.
