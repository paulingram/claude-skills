## Why

The `architect-team` pipeline ships strong interaction-correctness machinery at Phase 5 — the `interaction-completeness` team enumerates every interactive element + page, classifies each by wiring, and audits Playwright authenticity against the **live running app**. That catches drift after it has been built. It is much too late for one whole class of work:

The team that authors the OpenSpec proposal at Phase 0, and the planning gate at Phase 1, get no help reasoning about *which design elements MUST be interactive and what each one does* before they write the spec. The proposal can omit a "delete row" action that the design clearly implies; the spec can describe a "save" button without naming the endpoint that should receive its payload; the coverage map can list a route without committing to the per-element wiring. Each of those omissions costs a full Phase 5 cycle (or worse, a user-reported gap after merge) to discover.

The pipeline already produces three artifacts that, taken together, *almost* answer the question — `ROUTE_MAP.md` enumerates every interactive element in the existing code, `DESIGN_MAP.md` captures per-screen visual specs from the design source, and `INTEGRATION_MAP.md` enumerates the API surface across codebases. What is missing is the explicit cross-walk: **for each element on each designed screen, what action does it take, and which endpoint (existing or proposed) is it likely to call?**

The user's directive, verbatim:

> "lets have that same level of phase 5 intuiting for interactions as part of the phase 1 discovery. we need to make sure we analyze all designs so that when we do our route map and link the API to the front end, we have areas we know must be interactive and have guesses as to what they do. we need to present the user with a list of the mapping elements we have questions about (where its not sufficiently clear) and as a list, the user can sleect correct or not, then we ask about the ones we werent correct on."

The shape of the fix is the cross-walk artifact, the agent that produces it, and a **bulk user-verify gate** at the seam between Phase −1 and Phase 0 — present every low-confidence intuition as a single numbered list, the user marks the ones that are wrong, and then we drill down only on those. Items the user does not flag are auto-confirmed. The result feeds Phase 0 (spec authoring) and Phase 1 (coverage-map criteria) as a binding input.

## What Changes

- **Add** the `interaction-intuition` skill — a discovery-phase verification discipline modeled on the proven `interaction-completeness` skill, but pre-implementation and design-driven. For each frontend codebase, it cross-walks `ROUTE_MAP.md` × `DESIGN_MAP.md` × `INTEGRATION_MAP.md` (or the candidate-API surface) and produces an `INTERACTION_INTUITION_MAP.md` carrying every element on every designed screen with: intuited action, candidate endpoint(s), confidence, evidence, and an explicit ambiguity question when applicable. (REQ-001)
- **Add** the `interaction-intuiter` agent — spawned per frontend codebase during Phase −1B, immediately after `route-mapper` finishes producing `ROUTE_MAP.md` (and `DESIGN_MAP.md` when design inputs are present). Reads the maps, enumerates every interactive-by-design element, classifies each, names the candidate endpoint(s), assigns a confidence, and emits the per-codebase intuition map. Analysis-only; the only file it writes is the intuition map. (REQ-002)
- **Add** the `INTERACTION_INTUITION_MAP.md` artifact + schema — sibling to `ROUTE_MAP.md` and `DESIGN_MAP.md`. Carries `last_intuited` frontmatter, a `confirmed: bool` (false until the Phase −1D gate runs and the user has signed off), and per-element fields: `route`, `element_id`, `element_label`, `intuited_action`, `candidate_endpoints[]`, `confidence: high|medium|low|unknown`, `evidence[]`, `ambiguity_question?`, `user_verdict?: confirmed|corrected|deferred-to-impl`, `correction_note?`. Mined to MemPalace `--room interaction-intuitions`. (REQ-003)
- **Add** the confidence-classification rubric — `high` (clear element label + an exact existing endpoint match + design-spec context all agree); `medium` (clear label OR exact endpoint match, but not both, OR multiple plausible endpoint candidates); `low` (unclear label OR no obvious endpoint candidate OR conflicting design / route signals); `unknown` (no candidate inference possible — the element exists in the design but neither the route table nor the API surface points to an action). Every `low` and `unknown` element MUST surface to the Phase −1D bulk-verify gate; `medium` items surface when the agent explicitly flags them. (REQ-004)
- **Add** the Phase −1D bulk-verify gate — a new pipeline step that fires at the END of Phase −1 (after the integration map converges; before Phase 0 normalization begins). The orchestrator gathers every `low` / `unknown` / agent-flagged-`medium` element across every frontend codebase's intuition map and presents them to the user **as a single numbered list** — each item showing route, element, intuited action, candidate endpoint(s), confidence, and the agent's specific ambiguity question. The user replies in ONE of three formats: `"all correct"` (auto-confirm everything), a comma- / space-separated list of item numbers that are NOT correct, or `"all incorrect"` (drill down on every item). Then a targeted drill-down round asks one focused follow-up per flagged item (`AskUserQuestion` with up to 4 options when the candidate set is small; free-form question otherwise) — the answer updates the entry's `user_verdict`, `correction_note`, and `confirmed_action` / `confirmed_endpoint`. Items the user did NOT flag are auto-`confirmed`. (REQ-005, REQ-006)
- **Add** the binding-input rule — once the Phase −1D gate completes and every flagged item is resolved, the `confirmed: true` intuition map becomes a binding input to Phase 0 and Phase 1. Phase 0 spec authoring (proposal / design / specs) MUST reflect every confirmed element → action → endpoint mapping verbatim — a proposal that contradicts a confirmed intuition without an explicit user override (recorded as a `superseded_by` field in the entry) is a Phase 1 gate failure. Phase 1 coverage-map criteria for every `frontend` or `both`-layer requirement include the confirmed wirings as explicit acceptance criteria — every confirmed element MUST have a Playwright user-flow test in the corresponding Phase 5 verification. (REQ-007)
- **Add** the domain-gate carve-out — the existing v0.9.20 "gates are opt-in" rule applies to pipeline-process gates (proposal-first pause, "do you want me to proceed", clarifying questions whose answer is obvious). The Phase −1D bulk-verify is NOT a process gate; it IS the user-confirmation step inside the deliverable, exactly analogous to the editability-completeness team's `ambiguous` attribute escalation. The skill body's `## Default mode of operation` section is amended to make this distinction explicit so the rule and the new gate do not appear to contradict each other. (REQ-008)
- **Modify** the pipeline + discipline wiring — `architect-team-pipeline` SKILL.md (Phase −1B invokes `interaction-intuiter` per frontend codebase after `route-mapper`; new Phase −1D fires the bulk-verify gate; Phase 0 + Phase 1 read the confirmed map); `intake-and-mapping` SKILL.md (the Phase −1B step is documented in the canonical intake order); `frontend-route-mapping` and `design-fidelity-mapping` SKILL.md files (their outputs are named as inputs to interaction-intuition); `agents/route-mapper.md` (notes its output feeds the next step). (REQ-009)
- **Add** pytest structural coverage — the new skill + agent + map schema register correctly; the pipeline wiring + intake-and-mapping references resolve; cross-consistency unaffected; the schema documentation contains every required field; the confidence rubric is parametrized. (REQ-010)
- **Document & release** as v0.9.21 — README, CHANGELOG, CODEBASE_MAP (21 skills, 18 agents), INTEGRATION_MAP, CLAUDE.md, version bump. (REQ-011)

No breaking change to a target project. The new step only fires when a frontend codebase is in scope. With no frontend in scope (e.g., this plugin's own self-mod) the step is a silent no-op.

## Capabilities

### New Capabilities

- `interaction-intuition-discovery`: a discovery-phase enforcement layer guaranteeing that BEFORE the OpenSpec proposal is written, the pipeline produces a per-frontend-codebase intuition map cross-walking every designed element to its likely action and candidate endpoint, with explicit confidence, then collects the user's bulk verification of every low-confidence intuition in a single round, with a targeted drill-down only for items the user flagged. Confirmed intuitions are a binding input to Phase 0 spec authoring and Phase 1 planning. Delivered via a new judgment-heavy intuiter agent (`interaction-intuition` skill + `interaction-intuiter` agent), a new shared artifact (`INTERACTION_INTUITION_MAP.md`), a new Phase −1B production step, a new Phase −1D bulk-verify + drill-down gate, and a binding-input rule from the confirmed map into Phases 0 + 1.

### Modified Capabilities

None. No existing spec's requirements change. The change strengthens the discovery phase that precedes every existing capability (`pipeline-polish-v023`, `python3-portability`, `project-email-notifications`, `ui-interaction-fidelity`) — but does not alter any of their requirements. The `ui-interaction-fidelity` capability remains the post-implementation verification net; this change adds the pre-implementation discovery net upstream of it.

**Affected systems:** future pipeline runs (after the plugin updates to v0.9.21) produce `INTERACTION_INTUITION_MAP.md` per frontend codebase at Phase −1B and fire the bulk-verify gate at the end of Phase −1, before any OpenSpec change is scaffolded.

**Reuse-first decision summary:** The Phase 5 `interaction-completeness` skill is the right *pattern* (judgment-heavy, multi-source enumeration, confidence-bearing, escalate-don't-guess) but the wrong *phase* and the wrong *consumer* — Phase 5 verifies built code; this change reasons about not-yet-built work from design + maps. So `interaction-intuition` is a justified build-new that REUSES the `interaction-completeness` skill's structure wholesale (independent enumeration, classification rubric, escalation-not-guessing rule, gap-as-solution-requirement loop). The intuiter agent EXTENDS the existing `route-mapper` discipline by chaining off its output rather than redoing it — no rework of route enumeration. The Phase −1D gate is genuinely new (no equivalent bulk-verify-with-drill-down exists in the pipeline today), but it follows the same `AskUserQuestion`-driven escalation pattern already used by `editability-completeness` for `ambiguous` attributes — just bulked. Every other change extends an existing file. The full Reuse Decision Log is in `design.md`.
