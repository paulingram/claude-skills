## Why

The plugin has three Playwright-adjacent disciplines today — `playwright-user-flows` (the authoring discipline), `interaction-intuition` at Phase −1D (predicts which controls do what BEFORE the build), and `interaction-completeness` at Phase 5 (verifies built controls work end-to-end against the live app). All three operate on *the system we are building*. None of them answers a different and increasingly common question:

> *"I am a [persona]. Here is what I am trying to do on [this live site]. Does the site let me do it — and the adjacent things a real [persona] would actually need — without breaking?"*

That question is the entire job of the new `ux-test-builder` capability. The user describes a persona, an objective, and a target site (a URL OR `--dev` for the project's dev environment with credentials). The plugin maps the site (fresh or freshness-checked, reusing the existing `intake-and-mapping` + `route-mapper` + `interaction-intuition` discipline), drafts a literal Playwright flow matching the user's exact ask, and then asks **what should the persona ALSO be able to do that the literal description missed?** Three `flow-explorer` agents each propose 10-15 additional flows that exercise adjacent site capability the persona would realistically need — *"the user said 'upload and check files' but the site has THREE upload paths and the parsed-out data is reviewable on 10 other pages, so flows touching all four upload entries + sampling the parsed-data pages belong in the test plan."* Their outputs aggregate + distill to a unique set, each becomes a distinct Playwright flow, and three `flow-executor` agents run every flow in parallel against the live target. Results pool; disagreements re-examine to consensus per the 3-cycle bounded convergence pattern; bugs become structured solution requirements that auto-route through the existing `bug-fix-pipeline` (v0.9.22).

The capability turns "I have a vague UX concern about this site" into "here are 47 executed Playwright traces, 4 documented bugs, and 4 bug-fix SRs already in the queue."

## What Changes

- **Add** the `ux-test-builder` skill (REQ-001) — a new orchestrator playbook with phases U0 → U9 that takes a persona + objective + target site, maps the site, drafts a literal flow, expands it via 3 flow-explorers, distills, executes via 3 flow-executors, resolves disagreements, and routes documented bugs into `bug-fix-pipeline`. Five non-negotiable disciplines: real-site testing (never mocked); 3-agent convergence at both expansion and execution; literal-first-then-expand (never skip the user's exact request); bug-route-not-just-document (every documented bug becomes a solution requirement); explorer-expansion-is-context-aware (the 3 explorers MUST find adjacent capabilities the literal description missed, not just rephrase the literal).
- **Add** the `/architect-team:ux-test` command (REQ-002) — invokes the skill. Accepts the same two input forms as `/architect-team` (folder OR plain-language prose). New flags: `--site <URL>`, `--dev` (use the project's dev environment from `design.md` `## Dev Environment`), `--credentials <env-var>` (the env var name holding the auth payload — NEVER passed inline), `--persona <description>` (or read from prose), `--objectives <text>` (or read from prose).
- **Add** the U0 intake schema (REQ-003) — credentials reference (env-var name only, never the secret itself), site URL (or `--dev` resolved against the project's `design.md`), persona description, objectives. Persisted at `<cwd>/.architect-team/ux-tests/<persona-slug>/intake.json`.
- **Add** the U1 site-mapping step (REQ-004) — reuses `intake-and-mapping`'s discipline verbatim. If `CODEBASE_MAP.md` / `ROUTE_MAP.md` / `DESIGN_MAP.md` / `INTERACTION_INTUITION_MAP.md` exist and are current per the freshness check, the existing maps are used; else they are produced/refreshed. The Phase −1D bulk-verify gate fires when the intuition map has low-confidence items the user must resolve before flow expansion can proceed.
- **Add** the U2 literal-flow draft (REQ-005) — the orchestrator authors ONE Playwright flow matching the user's described task verbatim (no expansion yet). This flow is the persona's literal request; it becomes flow #1 in the eventual unique set. Persisted at `<cwd>/.architect-team/ux-tests/<persona-slug>/literal-flow.spec.ts` + `.json` (the structured metadata).
- **Add** the U3 flow-expansion via 3 `flow-explorer` agents (REQ-006) — each agent reads the persona description + the literal flow + the full site maps and INDEPENDENTLY proposes 10-15 additional Playwright user-flow specifications. Each explorer is explicitly prompted to discover adjacent capabilities (additional entry points for the same action, alternate flows to the same outcome, related pages where the same data surfaces, settings / preferences a persona would adjust, multi-step workflows, etc.) — NOT to rephrase the literal. Each draft is persisted at `<cwd>/.architect-team/ux-tests/<persona-slug>/expansions/explorer-<N>-<ts>.json`.
- **Add** the U4 distillation step (REQ-007) — the orchestrator aggregates the 3 explorers' lists (3 × 10-15 = 30-45 raw proposals) and deduplicates semantically (not just string-match — two flows that cover the same user-experience are duplicates even with different selectors). Produces `<cwd>/.architect-team/ux-tests/<persona-slug>/distilled-flows.json` with the unique set; typically 15-25 flows after dedup. Each entry carries a `source_explorers[]` list crediting which explorer(s) proposed it.
- **Add** the U5 Playwright authoring step (REQ-008) — one `.spec.ts` per distilled flow, persisted at `<cwd>/.architect-team/ux-tests/<persona-slug>/playwright/<flow-N>.spec.ts`. Per `playwright-user-flows`: real `page.click` / `page.fill` / `page.waitFor` / `page.selectOption` / `page.press` / `page.setInputFiles` — NEVER `page.request.*`; NEVER mocked. Per-step expectations files written BEFORE the test runs (per `root-cause-test-failures`). Login uses the credentials env-var from U0.
- **Add** the U6 parallel execution via 3 `flow-executor` agents (REQ-009) — each executor runs every distilled Playwright flow independently against the live target site. The redundancy is the consensus mechanism: flaky tests, intermittent UI states, race conditions, and environment dependencies surface as disagreements rather than passing through silently. Each executor's per-flow result is persisted at `<cwd>/.architect-team/ux-tests/<persona-slug>/executions/executor-<N>/<flow-N>.json` with `verdict: pass | fail | flaky | env-failure` + the captured trace + the captured screenshot(s) + the per-step expectation deltas.
- **Add** the U7 consensus-on-disagreements step (REQ-010) — the orchestrator pools the 3 executors' per-flow verdicts. Flows where all 3 agreed (typically 80-90%) are recorded as the consensus verdict. Flows where the 3 disagreed enter a re-examination loop: each executor re-runs the disputed flow with the OTHER executors' verdicts + traces as additional context (mirroring the editability-completeness / interaction-completeness Round 2 round-robin). Up to 3 re-examination cycles. If consensus still can't be reached, the orchestrator escalates to the user with the divergent verdicts as evidence (a domain gate, per v0.9.21 — fires regardless of `--proposal-first`).
- **Add** the U8 bug-routing step (REQ-011) — every flow with a final consensus verdict of `fail` becomes a structured bug artifact at `<cwd>/.architect-team/ux-tests/<persona-slug>/bugs/<bug-N>.json` with the failing flow, the captured trace + screenshot(s), the persona description, the literal-vs-actual symptom. Each bug is then promoted to a solution requirement with `origin.kind: "ux-flow-failure"` and auto-routes through the existing v0.9.22 `bug-fix-pipeline` (the orchestrator dispatches the bug-fix pipeline against each bug; the bug-fix pipeline's own discipline — replicate, propose, fix, QA-replay — applies). The UX test builder run does NOT block on bug fixes; the bugs are queued, and the run's final report includes the bug-fix-pipeline dispatch references.
- **Add** the U9 final report (REQ-012) — summary: persona, objectives, site, # flows attempted, # passed, # failed, # disagreements resolved / escalated, list of bugs with their bug-fix-pipeline dispatch SHA / branch references. Auto-committed and auto-pushed per the same Phase 8 default-branch guard discipline.
- **Add** the new agents (REQ-013):
  - `flow-explorer` (×3, opus, bounded `Write` to the expansions directory) — reads persona + site maps + literal flow; proposes 10-15 additional flows.
  - `flow-executor` (×3, opus, `Bash` + `Write` to the executions directory) — runs Playwright flows against the live target; documents outcomes.
- **Add** pytest structural coverage (REQ-014) — the new skill / agents / command register correctly; the 10 phases U0-U9 are documented; the disciplines are stated; the bug-fix-pipeline integration is wired.
- **Add** the `bug-fix-pipeline` Phase B6b Logical Sensibility Check (REQ-016) — a NEW phase between B6 (QA replay) and B7 (Archive + Report) that closes the gap surfaced by a real-world user report: *"The fix correctly routes Sign Back In to /login, but the deployed bundle is hermetic (no VITE_\* baked), so the LoginScreen shows 'auth-unavailable.' Let me investigate and fix. There needs to be a post deployment check for sensibility on all elements touched."* B6's QA-replay verifies the ORIGINAL symptom is gone — necessary but not sufficient. The fix's diff may have touched (directly or transitively) UI components, routes, API endpoints, and dependent flows that the original symptom check doesn't exercise. B6b reads the fix's git diff, computes the **impact set** (every component the fix touched + every page that imports those components + every route that references those components + every API endpoint the fix touched), and dispatches the new `fix-sensibility-checker` agent to author + run minimal Playwright sensibility flows against each item in the impact set, against the deployed dev environment. Each sensibility flow asserts the item renders and behaves logically — no error banners ("auth-unavailable" / "service-unavailable" / "configuration-missing"), no console errors, no broken auth surfaces, no 404/500 responses on dependent navigations, no missing-data placeholders where data should be present. Any surfaced issue becomes a fresh solution requirement (`origin.kind: "fix-regression"`) and routes back through the bug-fix-pipeline (recursive — a new bug-fix loop). The current fix is NOT marked complete until B6b returns clean AND B6 has already returned `bug-resolved`.
- **Add** the `fix-sensibility-checker` agent (REQ-017) — opus, bounded `Write` to `.architect-team/sensibility/<bug-slug>/`; `Bash` for Playwright execution. Reads the fix's diff + the impact-set-computation rules + the deployed dev URL + the credentials env-var. Computes the impact set (UI components + their importers + their routes + their endpoints). For each item in the impact set, authors a minimal Playwright sensibility flow + runs it against the deployed dev environment. Documents each item's verdict (`sensible | nonsensical | env-failure | not-reachable`). Writes a structured report; the orchestrator routes any `nonsensical` items to the bug-fix-pipeline as fresh SRs.
- **Document & release** as v0.9.29 (REQ-018) — README banner / NEW IN panel / inventory grid (skills 22→23, agents 22→25, commands 7→8) / timeline; CHANGELOG; CODEBASE_MAP; CLAUDE.md; INTEGRATION_MAP; plugin.json; marketplace.json.

No breaking change. The new ux-test-builder capability is reachable ONLY via the explicit `/architect-team:ux-test` command; the bug-fix-pipeline's new Phase B6b is purely additive (it tightens the verification gate; it doesn't change when B7+B8 fire or what they do). The main `/architect-team` pipeline is unchanged.

## Capabilities

### New Capabilities

- `ux-test-builder`: a persona-driven UX test orchestrator. Given a persona description + objectives + target site (URL or `--dev`) + credentials, it (1) maps or refreshes the site, (2) drafts a literal Playwright flow matching the user's request, (3) expands the flow set via 3 independent explorer agents discovering 10-15 adjacent flows each, (4) distills to a unique set, (5) authors one Playwright `.spec.ts` per unique flow, (6) executes every flow via 3 parallel executor agents against the live target, (7) resolves verdict disagreements via the standard 3-cycle bounded convergence, (8) documents every failed flow as a bug + routes it through the existing `bug-fix-pipeline`, and (9) emits a final report. The capability is the structural bridge between "human describes a persona's UX concern" and "the existing bug-fix-pipeline has the work queued."

### Modified Capabilities

- `bug-fix-pipeline` (v0.9.22) — gains TWO additions:
  1. A new SR origin kind (`ux-flow-failure`) that the U8 bug-routing step uses to enqueue UX-test-discovered bugs.
  2. A new Phase B6b (Logical Sensibility Check) between B6 (QA replay) and B7 (Archive + Report) that closes a real-world gap: the QA-replay confirms the ORIGINAL symptom is gone, but the fix may have touched elements / pages / endpoints that the original symptom check doesn't exercise. B6b uses the new `fix-sensibility-checker` agent to compute the impact set from the fix's git diff and run minimal Playwright sensibility checks on every item in the impact set against the deployed dev environment. Any surfaced issue becomes a fresh SR with `origin.kind: "fix-regression"` and routes back through the bug-fix-pipeline.
  
  The bug-fix-pipeline's other disciplines (replicate → reproduce-test → propose → fix → QA-replay → archive → commit) are unchanged. B6b is purely a tightening of the verification gate.

**Affected APIs / dependencies:** none new. The capability uses the EXISTING Playwright + chromium stack (already a setup.py dependency), the existing `intake-and-mapping` / `route-mapper` / `interaction-intuition` skills, and the existing `bug-fix-pipeline` for downstream bug resolution.

**Affected systems:** future `/architect-team:ux-test` invocations against any live target site (URL or `--dev`) produce a persona-driven test plan, execute it 3 times in parallel against the live site, document bugs, and queue them in the bug-fix-pipeline. The UX test builder is a separate command from `/architect-team` and `/architect-team:bug-fix` — neither of those is changed.

**Reuse-first decision summary:** The capability is built almost entirely from reused skills + agents. `intake-and-mapping` + `route-mapper` + `interaction-intuition` provide the site map (no new mapping code). `playwright-user-flows` provides the Playwright authoring discipline. `root-cause-test-failures` provides the expectation-file + 3-pass RCA discipline for flow failures. `bug-fix-pipeline` provides the downstream bug resolution (new SR origin kind `ux-flow-failure` is the only API surface change). The 3-agent convergence pattern (3 explorers, 3 executors, round-robin consensus) is REUSED from `editability-completeness` / `interaction-completeness` / `integration-explorer`. The new skill is justified build-new (a new orchestration shape — persona + literal + expand + distill + execute + arbitrate + route — that doesn't fit into any existing skill). The two new agents are justified build-new (each is a discrete role: propose flows / execute flows). Full Reuse Decision Log in `design.md`.
