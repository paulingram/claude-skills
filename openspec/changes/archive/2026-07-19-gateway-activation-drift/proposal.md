# Proposal — gateway-activation-drift

## Why

Activation drift in the CT6 external-LLM gateway is undetected and un-healed. "Activation" is exactly two env entries (`ANTHROPIC_BASE_URL` + `ANTHROPIC_AUTH_TOKEN`) that `scripts/setup/install_gateway.py` merges into `~/.claude/settings.json` on `--activate`. Observed on the owner's machine 2026-07-18→19: something rewrote settings.json non-merge-preservingly and dropped the env block while `~/.architect-team/gateway/gateway.json` still recorded `activated: true`. A subsequent `setup.py --external-llm` run reported **"activated=carried-forward; CONFIRMED serving live"** — every Claude Code session launched in that window silently ran direct-to-Anthropic with the secondary split off, while every status surface stayed green.

Replication evidence (`tests/bug-fix-gateway-activation-drift/test_replication.py`, executed 2026-07-19, `5 failed in 3.47s`, sandboxed via `--base-dir` + `--settings-path` + `CT6_GATEWAY_HOME`):

> FACET-A `test_status_surfaces_activation_drift_explicitly` — "status output does NOT surface activation drift explicitly — recorded activated=True while settings.json env block is absent, but the surface reports only a bare `activated=False` / `not applied` with no drift/mismatch qualifier."
> FACET-A `test_status_cli_subprocess_end_to_end_pins_the_report_surface` — REAL CLI subprocess, same green-lie on the printed report surface.
> FACET-B `test_carry_forward_verifies_claude_env_against_settings` — "install carry-forward path reports green (activated=carried-forward) from recorded state alone — never verified claude_env_applied() against the given --settings-path, which is drifted."
> FACET-C `test_sessionstart_has_activation_heal_seam` — "SessionStart hook has NO activation-heal seam. maybe_heal_model_split heals the model split but nothing heals the settings.json env block when gateway.json says activated=true and the block is absent."
> FACET-C `test_sessionstart_main_does_not_restore_drifted_env` — "SessionStart main() did NOT restore the drifted env block — the hook has no activation-heal path."

Root causes, per the executed light-discriminant (verdict: logic-layer bug; `.architect-team/bug-isolation/gateway-activation-drift/discriminant.json`):

- **(A) status** — `_cmd_status` computes `report.activated = claude_env_applied(...)` honestly (`install_gateway.py:3397`) but never compares it against the RECORDED `state.get("activated")`, so a drifted machine prints a bare `activated=False` / "Claude Code activation: not applied" (`:3408-3414`, footer `:3858`) indistinguishable from a machine that was never activated. The contradiction — the signal that something silently destroyed the wiring — is swallowed.
- **(B) install carry-forward** — the `elif prior_state.get("activated"):` branch (`install_gateway.py:3201-3209`) emits an ok row ("activation carried forward from the prior install") from recorded state alone; the state-write (`:3364`) ORs `prior_state.get("activated")` back in; `setup_entry` (`:3661-3677`) then prints `activated=carried-forward` + "CONFIRMED serving live". The gateway-side live confirmation cannot see the client-side wire, so the whole surface is green while Claude Code is unpointed.
- **(C) SessionStart** — `hooks/sessionstart-run-continuity.py` heals model-split drift (`maybe_heal_model_split`, `:136-235`) but has NO symmetric activation heal; `main()` (`:255`) invokes only the split heal. The drift persists across every session restart.

## What Changes

- **Drift detection on every activation-reporting surface.** `_cmd_status` gains a drift computation (recorded `activated` truthy AND `claude_env_applied()` false) and surfaces it loudly: the summary line reports `activated=DRIFTED(...)`-class text, a dedicated drift step row names the recorded-vs-settings contradiction + the remediation, and the footer prints a drift qualifier instead of the generic "not applied". A plain not-activated machine's output is UNCHANGED (drift text must never appear on clean machines).
- **The install carry-forward path verifies, then heals.** The `elif prior_state.get("activated"):` branch verifies `claude_env_applied()` against the resolved settings path. Verified → keep the ok carry-forward row (now factually true). Drifted → re-apply the env block via the existing `apply_claude_env()` from the RECORDED port + the PERSISTED master key in `gateway.env` (recorded `activated: true` IS the prior consent; no re-derivation of the key), report "activation drift healed — env block re-applied"; when the master key cannot be resolved, emit a FAIL row naming the drift + the `install --activate` remediation — never a green carried-forward. `setup_entry`'s `activated=` display follows the new report fields.
- **SessionStart activation self-heal.** New `maybe_heal_activation()` in `hooks/sessionstart-run-continuity.py`, symmetric to `maybe_heal_model_split`: same installed-plugin-copy guard (with an explicit-injection bypass — injecting BOTH `gateway_state_path` + `settings_path` is programmatic consent to those sandbox paths; the real `main()` wiring never injects), fires when gateway state records `activated` + `enabled` + `auth_mode: api-key` AND settings.json lacks `ANTHROPIC_BASE_URL`; verifies the gateway is actually LISTENING on the recorded port before writing (a short injectable TCP liveness probe — the recorded `enabled` flag is never trusted bare, per the B4 audit: the audited class must not recur inside the fix); re-applies the env block merge-preservingly from the recorded port + persisted master key; NEVER clobbers a present-but-different `ANTHROPIC_BASE_URL` (a user-customized value wins, mirroring `remove_claude_env`'s posture); NEVER overwrites an existing-but-corrupt settings.json; fail-open (every failure path returns `""`); prints a one-line heal note incl. the restart caveat. `main()` invokes it alongside the split heal.
- **Regression tests**: the B1 replication tests flip green (with the B4-sanctioned FACET-C artifact repair by their author — the original `main()`-level test was unsatisfiable on Windows); new unit tests pin the heal's guards (explicit-injection bypass, dead-port no-op, custom-BASE_URL never clobbered, corrupt-settings abort on BOTH heals, fail-open on missing/corrupt state or gateway.env, merge-preservation of unrelated settings keys, plain-not-activated output unchanged, carry-forward verification against the served port).

- **REQ-004 — the root clobberer (found mid-run): the repo's own test suite deactivates the real machine.** `tests/test_install_gateway.py::test_uninstall_purge_removes_state_dir` (`:855-860`) invokes a REAL uninstall with `--base-dir`/`--agents-dir` sandboxed but NO `--settings-path`; `_cmd_uninstall` (`install_gateway.py:3491-3493`) then resolves the DEFAULT real `~/.claude/settings.json` and `remove_claude_env()` strips `ANTHROPIC_BASE_URL`+`ANTHROPIC_AUTH_TOKEN` merge-preservingly whenever the real BASE_URL matches port 4000. On CI/clean machines it silently no-ops; on the owner's activated machine EVERY full suite run deactivates the gateway — this is the root cause of BOTH observed drift incidents (2026-07-18→19: the v3.41.0 run's seven suite iterations; 2026-07-19 ~10:04Z: this run's own B5 suite runs, mtime-correlated). Fix, class-scoped: (a) the leaking test gains `--settings-path`; (b) an autouse fixture in `tests/test_install_gateway.py` redirects the module's `DEFAULT_USER_SETTINGS_PATH` to a per-test sentinel so NO test in the module can ever reach the real file, with a probe test proving absorption; (c) a session-scoped tripwire in `tests/conftest.py` snapshots the real `settings.json` + `gateway.json` + `gateway.env` at suite start and fails LOUDLY at suite end on any mutation — converting any future leak of this class, in any test file, into a named suite failure; (d) `tests/test_install_librarian.py:347` (`install --json` with no `--base-dir`) audited and sandboxed if leaky.

- `scripts/setup/install_gateway.py` — `_cmd_status` (drift row + summary + footer), the install carry-forward branch, `setup_entry` display mapping. Existing helpers `claude_env_applied` / `apply_claude_env` / `read_env_file` are REUSED, not duplicated.

- `tests/bug-fix-gateway-activation-drift/` (B1 artifacts flip green) + new heal/guard unit tests; suite stays green under Windows cp1252 and `PYTHONUTF8=1`.
- `CHANGELOG.md`, `README.md` (gateway section: drift detection + self-heal note), version bump 3.41.0 → 3.41.1 (PATCH — defect fix, no new selectable surface).
- NOT touched: the LIVE machine gateway (no config regeneration, no restart — the fix is source + hook logic only), archived changes, the model-split heal's behavior.
