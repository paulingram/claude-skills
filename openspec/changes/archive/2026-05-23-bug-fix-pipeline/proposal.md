## Why

The `architect-team-pipeline` is excellent for greenfield features and substantial new capability work, but it carries weight every run: the 100%-coverage Phase 1 gate, Phase 2's parallel team spawn, Phase 5's six independent review-and-converge teams, the master-review audit at Phase 7. For a 30-line bug fix — *"the row-action menu's Delete button doesn't actually delete; clicking it just closes the menu"* — the full pipeline is the wrong tool. The investment is correct for a new feature; for a known-bug-with-a-clear-symptom it is a tax that delays a fix users are already waiting for.

What that tax masks is that bug fixes have their *own* discipline that the main pipeline only loosely captures:

1. **Replicate first.** A fix proposed without a replication is a guess. For frontend bugs, the replication is a Playwright user-flow the agent drives BEFORE proposing a fix; for backend bugs, it is a script that exercises the failing path against the live dev API. If the symptom can't be reproduced, the bug isn't understood.
2. **Reproduction is the regression test.** The Playwright flow / backend script that just demonstrated the bug becomes the test the QA replay confirms against post-fix — there is no separate "now write the test" step. For frontend bugs the agent ALSO writes a backend diagnostic test (the bug experienced on the frontend is verified through the backend's view of the same flow), so the QA replay covers both layers and a regression that only one layer catches is impossible.
3. **Generalize, don't patch the symptom.** A fix for *"this one user's row doesn't delete"* that special-cases that user's ID is not a fix — it is a debt deposit. The architect's job after the proposal is to confirm the fix addresses the *class* of bug, not the specific instance. The discipline is non-negotiable except when the user explicitly authorizes a one-off (a hot-patch before a public event, etc.).
4. **QA replays against the live dev environment, not unit tests.** The bug is in the running system; the verification has to be in the running system. After the fix lands and deploys to the dev environment, the QA agent re-runs the original replication flow / script against the deployed dev URL — same Playwright, same backend script. Pass = bug gone. Fail = back to architect with the new evidence; a fresh OpenSpec proposal; a fresh deploy; a fresh QA replay. Loop until the bug is resolved entirely — not "the test passes again," but the originating symptom is gone end-to-end.
5. **Auto-routing when the main pipeline is invoked with bug-shaped work.** A user who types `/architect-team` and describes a bug shouldn't have to know there's a separate command — the main orchestrator should triage the intake, recognize a bug, and route the bug-portion to the bug-fix path. When the requirement is mixed ("fix the heir-totals percentage and also add a new export button"), the orchestrator splits the work and runs the bug-fix sub-pipeline in parallel with the feature pipeline, so the bug fix doesn't wait for the feature build.

## What Changes

- **Add** the `bug-fix-pipeline` skill (REQ-001) — a faster bug-focused orchestrator playbook. Phases B−1 → B8 mirror the main pipeline's discipline at the structural points (intake + maps freshness, OpenSpec proposal authoring, architect review, deploy, archive + push) while replacing the heavy parallel-team-spawn / 6-team Phase 5 with a tight **replicate → reproduce-test → propose → fix → QA-replay loop**. Iteration ceiling: the same global 20-step ceiling + oscillation detection from `architect-team-pipeline` applies; on bug-fix work the bound is typically reached at 4-6 iterations.
- **Add** the `/architect-team:bug-fix` command (REQ-002) — invokes the `bug-fix-pipeline` skill. Accepts the **same two input forms as `/architect-team`**: a requirements folder OR a plain-language requirement typed directly as prose (a bug report, a symptom description, a screenshot reference). Both first-class; the command's argument-parsing rules mirror `/architect-team` exactly — never refuses prose, never treats the first word of a sentence as a path, never asks "give me a requirements folder."
- **Add** the freshness pre-scan as the first phase of the bug-fix pipeline (REQ-003) — `bug-fix-pipeline` Phase B−1 reuses the `intake-and-mapping` discipline verbatim. The maps it consumes (CODEBASE_MAP / ROUTE_MAP / DESIGN_MAP / INTEGRATION_MAP / INTERACTION_INTUITION_MAP per frontend codebase) must be as fresh as the most recent commit — same rules, no shortcut. A bug fix proposed against a stale map is the second-worst class of bug fix (after one proposed without replication).
- **Add** the bug-replication discipline (REQ-004) — Phase B1 is *replicate first, then propose*. For frontend bugs the orchestrator spawns the `bug-replicator` agent (REQ-011), which reads the bug description, identifies the user flow that should reproduce it, and writes + runs a Playwright user-flow against the live dev environment (per `playwright-user-flows`). For backend bugs the agent writes + runs a script (curl / httpx / a small Python harness) against the live dev API (per `dev-api-integration-testing`). When the bug description is ambiguous — the user hasn't named the screen, or the steps, or the input — the replicator stops and asks the user a structured clarifying question ("How did you experience the bug? What did you click? What did you expect to see vs. what you saw?"). It does NOT guess.
- **Add** the reproduction-artifact rule (REQ-005) — the Playwright user-flow / backend script that demonstrated the bug IS the regression test. Phase B2 promotes the replication artifact into the codebase's test suite (the `tests/` or `e2e/` directory of the affected codebase). For frontend bugs, Phase B2 ALSO authors a **backend diagnostic test** that exercises the same flow from the backend's perspective (a Python / Node script that calls the endpoint(s) the Playwright flow drove and asserts the data-layer outcome), so a regression that only one layer catches is impossible. For backend-only bugs the script alone suffices.
- **Add** the OpenSpec proposal step (REQ-006) — Phase B3 authors a slim OpenSpec change (`openspec/changes/<bug-slug>/`) with the same artifact chain as the main pipeline (`proposal.md`, `design.md`, `specs/<cap>/spec.md`, `tasks.md`, `coverage-map.json`). The proposal cites the replication evidence (the Playwright trace / script output) verbatim as the failure-mode source, names the root cause if known from the replication, and proposes the fix. The Phase 1 planning-validation loop runs against this proposal exactly as it does for a feature change — the discipline is the same; only the team-spawn and Phase 5 review-team-stack are replaced by the bug-fix-specific loop.
- **Add** the generalized-fix architect review (REQ-007) — Phase B4 dispatches the `system-architect` agent in a new **Bug-Fix Generalization Audit** mode. Inputs: the bug description, the replication artifact, the proposed fix in `design.md`, the relevant CODEBASE_MAP + INTEGRATION_MAP. Outputs a verdict (`pass | needs-generalization | needs-replacement`) with reasoning. **A "fix" that special-cases the specific failing input — a literal user-id in a conditional, a hard-coded category name in a switch, a localized patch where the underlying logic is broken for a class of inputs — is REJECTED.** The override is explicit: if the user instructed the run *"just hard-code it for now"* (or equivalent), the architect records the authorization verbatim in the verdict and lets the targeted fix proceed. Otherwise generalization is mandatory.
- **Add** the QA-replay loop (REQ-008) — Phase B6 dispatches the new `qa-replayer` agent (REQ-012) after the fix is implemented and deployed. The QA replayer's input is the reproduction artifact from Phase B2 (the Playwright flow + the backend diagnostic, OR the backend script alone). It re-runs them against the live dev environment, verbatim — no edits, no shortcuts. **Pass criterion: the originating symptom is gone end-to-end.** Not "the test passes" — the original failure mode is no longer reproducible. On fail, the replayer writes a fresh solution requirement with the new evidence and routes back to Phase B3 (new OpenSpec proposal); the loop continues, bounded at 10 bug-fix iterations (the global 20-step ceiling still caps it, but bug fixes earn a tighter local bound — most bugs should be one or two cycles; ten is already a smell).
- **Add** the live-dev-environment-by-default rule (REQ-009) — Phase B5 ALWAYS deploys the fix to the dev environment (per the target project's `design.md` `## Dev Environment` section) BEFORE Phase B6 testing. The fix is verified in the running system, not locally. The exception, and the ONLY exception, is when the run was explicitly invoked with `--environment production` (or the user's prose names the target as production) — in which case the orchestrator escalates a structured question (production deploys are user-decisions, not orchestrator decisions) and does not deploy automatically. Builds are confirmed green BEFORE the QA replay starts; a failed build is a Phase B5 escalation that routes back to the implementing team, not a QA-replay failure (the bug-fix loop should never confuse a deploy failure with a fix failure).
- **Add** the `/architect-team` triage / dispatch layer (REQ-010 + REQ-013) — the main `architect-team-pipeline` skill gains a new Phase −2 (just before the existing Phase −1 prelude) that **classifies the incoming requirement** as `bug | feature | mixed | unclear`. The classification is performed by the new `bug-classifier` agent (REQ-013) — read-only, analysis-only — from the source description's language signals (bug-keywords vs. feature-keywords vs. both) PLUS optional explicit overrides (the user said "/architect-team:bug-fix" → forced `bug`; the user typed `--bug-fix` → forced `bug`). The orchestrator then routes:
  - **Pure-bug** → invoke the `bug-fix-pipeline` skill against the requirement; do NOT continue the architect-team-pipeline phases.
  - **Pure-feature** → continue the existing architect-team-pipeline phases as before (no behavior change for feature work).
  - **Mixed** → split the requirement into a bug-portion and a feature-portion (the classifier provides both); spawn TWO subagents IN PARALLEL (one running `bug-fix-pipeline` against the bug-portion, one running `architect-team-pipeline` against the feature-portion); orchestrator awaits both and integrates their commit ranges in the final report. The user can override the auto-split (e.g., *"do them sequentially — the bug fix is in the same file the feature touches"*).
  - **Unclear** → the classifier emits a structured question to the user ("Is this a bug to fix, a new capability to build, or both?") and pauses. This IS a domain gate (per v0.9.21's process-vs-domain-gate carve-out) — the user's answer changes what is built.
- **Add** the same-input-forms guarantee (REQ-002 + REQ-010) — the `bug-fix-pipeline` skill body, the `/architect-team:bug-fix` command body, AND the `architect-team-pipeline` Phase −2 triage logic all enforce the same input-form rules: a requirements folder OR a plain-language prose requirement; both first-class; never refuse prose; never path-treat the first word; ask the user only when the input is genuinely empty.
- **Add** the `bug-replicator` agent (REQ-011) — opus, analysis + bounded write access (writes the reproduction artifact files into the target codebase's tests directory, nothing else). Reads CODEBASE_MAP + ROUTE_MAP + INTEGRATION_MAP + the source description; spawns a single Playwright session OR a backend script and runs it; reports `reproduced | could-not-reproduce | needs-clarification` with evidence.
- **Add** the `qa-replayer` agent (REQ-012) — opus, read-only on source + bounded execution (re-runs the reproduction artifacts via Bash; never edits feature code). Reports `bug-resolved | bug-still-present | env-failure` with evidence; on `bug-still-present` writes a solution requirement back to the orchestrator.
- **Add** the `bug-classifier` agent (REQ-013) — sonnet (lighter — pure classification, no judgment-heavy work), analysis-only (Read / Grep / Glob only; no Bash, no Edit, no Write). Returns a structured verdict: `{ kind: bug|feature|mixed|unclear, bug_portion: <text or null>, feature_portion: <text or null>, confidence: high|medium|low, reasoning: <one-line citation of the language signals >}`.
- **Add** pytest structural coverage (REQ-014) — the new skill + 3 agents + command register correctly; the bug-fix-pipeline body has every named phase B−1..B8; the architect-team-pipeline body has the new Phase −2 triage section; intake-and-mapping has the matching dispatcher reference; all input-form rules covered.
- **Document & release** as v0.9.22 (REQ-015) — README banner, NEW IN v0.9.22 panel, timeline, inventory grid (22 skills, 21 agents, 7 commands); CHANGELOG; CODEBASE_MAP; CLAUDE.md; INTEGRATION_MAP; plugin.json; marketplace.json.

No breaking change to a target project. The new bug-fix pipeline is opt-in via `/architect-team:bug-fix` (explicit) or auto-routed via `/architect-team` (when the classifier returns `bug` or `mixed`). For feature work the main pipeline behavior is unchanged.

## Capabilities

### New Capabilities

- `bug-fix-pipeline`: a faster, bug-focused orchestrator with five non-negotiable disciplines — **replicate first** (Playwright for frontend / backend script for backend / ambiguity-escalation when unclear); **reproduction is the regression test** (the artifact that demonstrates the bug becomes the QA-replay artifact; frontend bugs ALSO author a backend diagnostic); **generalized fix** (the `system-architect` Bug-Fix Generalization Audit rejects symptom patches unless the user explicitly authorized one); **QA replay against the live dev environment** (real deploys, real URLs, the QA agent re-runs the reproduction artifacts and the pass criterion is "the originating symptom is gone end-to-end"); **live-dev-environment-by-default** (fixes deployed to the dev environment URLs before testing; production is an explicit-opt-in exception that escalates). The whole loop is bounded — 10 bug-fix iterations as the local ceiling, the existing global 20-step ceiling as the absolute. Triggered by `/architect-team:bug-fix` (explicit) or by the main `/architect-team` triage when the `bug-classifier` returns `bug` or `mixed`.

### Modified Capabilities

- `pipeline-polish-v023` — the main `architect-team-pipeline` skill gains a new Phase −2 (Triage & Routing) that classifies the incoming requirement and either routes it to `bug-fix-pipeline`, continues to the existing Phase −1 (feature work), OR spawns both in parallel for `mixed` requirements. Existing Phase −1 → 8 behavior is unchanged for `pure-feature` runs. No requirement of `pipeline-polish-v023` changes; the triage layer is additive.
